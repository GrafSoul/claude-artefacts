Docker — ментальная модель

Не про синтаксис — про картинку в голове. Что такое образ и чем он отличается от контейнера, почему сборка бьётся на слои и когда они берутся из кэша, зачем тома и сети. Пощупай интерактив. А построчный разбор Dockerfile и compose — рядом, в справочнике.

Зачем вообще Docker

Классическая боль: «у меня работает, а на сервере — нет». Причина — разное окружение: версии Node, библиотеки, настройки ОС. Docker упаковывает приложение вместе со всем его окружением в изолированную коробку — контейнер. Эта коробка одинаково запускается на твоём ноутбуке, у коллеги и на боевом сервере.

📦 что внутри коробки

Приложение + нужная версия Node + все зависимости + минимальный Linux. Всё, что нужно для запуска, — с собой.

🧩 чем не виртуалка

Контейнер лёгкий: он делит ядро ОС хоста, а не тащит целую операционку. Стартует за секунды, весит мегабайты, а не гигабайты.

Образ vs контейнер — главная развилка

Эти два слова путают чаще всего. Образ — это слепок, рецепт: неизменный шаблон приложения. Контейнер — запущенный экземпляр образа, живой процесс. Из одного образа можно запустить сколько угодно контейнеров. Нажми docker run и посмотри.

📀 myapp:latest образ — неизменный слепок, один на всех
аналогия

Образ — это класс, контейнер — объект. Или: образ — рецепт торта, контейнеры — испечённые по нему торты. Рецепт один, тортов много, и каждый живёт своей жизнью.

Слои и кэш — почему порядок важен

Образ собирается по Dockerfile сверху вниз, и каждая инструкция создаёт слой. Docker кэширует слои: если инструкция и всё до неё не менялись — слой берётся из кэша мгновенно. Как только что-то изменилось — этот слой и все ниже пересобираются заново.

Нажми, что ты изменил, и посмотри, какие слои уцелеют в кэше, а какие пойдут на пересборку.

вот зачем это знать

Поэтому в Dockerfile COPY package*.json и RUN npm ci идут раньше, чем COPY . .. Правки кода (частые) не сбрасывают кэш установки зависимостей (медленной) — она пересобирается, только когда реально поменялся package.json. Порядок инструкций = скорость сборки.

Тома — данные живут отдельно

Контейнер эфемерный: удалил и пересоздал — внутри всё как в свежем образе, изменения пропали. Это хорошо для приложения (всегда чистый старт), но плохо для данных базы. Решение — том: хранилище, которое живёт вне контейнера и переживает его пересоздание.

🗄️ именованный том

pgdata, redisdata — Docker хранит их сам, отдельно от контейнера. Снёс контейнер базы — данные в томе остались.

🔗 bind-mount

Папка с хоста, «проброшенная» внутрь (.:/app). Основа hot-reload: правишь код на машине — контейнер сразу видит.

запомни различие

Контейнер — эфемерный, том — постоянный. Всё, что нельзя потерять, должно лежать в томе, а не «внутри контейнера». Именно поэтому у базы и кэша есть volumes.

Сети — как контейнеры находят друг друга

Compose кладёт все сервисы в общую приватную сеть, где они видят друг друга по имени сервиса. Поэтому backend подключается к базе по адресу postgres, а не localhost или IP — Docker сам разрешает имя в нужный контейнер.

📛 имя = адрес

postgres, redis, backend — это и имена сервисов, и хосты внутри сети. IP знать не нужно.

🚪 изоляция

Снаружи виден только тот, кто сам опубликовал порт (ports). База без ports доступна лишь внутри сети — как в проде.

это ты уже видел

Ровно эта сеть нарисована на карте системы: в проде наружу торчит только Nginx, а postgres/redis живут в приватной сети по именам.

Словарь Docker

Шесть слов, на которых держится всё остальное. Синтаксис каждого — в справочнике.

Dockerfile

Рецепт образа. Список инструкций: из чего собрать, что скопировать, чем запускать.

образ (image)

Слепок по рецепту. Неизменный шаблон, из которого запускают контейнеры.

контейнер

Запущенный экземпляр образа — живой изолированный процесс с приложением.

слой (layer)

Результат одной инструкции Dockerfile. Кэшируется, чтобы не пересобирать неизменное.

том (volume)

Постоянное хранилище вне контейнера. Данные переживают его пересоздание.

compose

Оркестровка: как запускать сервисы вместе — порты, тома, сеть, связи — одним файлом.

две роли одной фразой

Dockerfile отвечает «как собрать образ», compose — «как запускать это вместе». Одно — рецепт, другое — оркестровка.

Частые вопросы

Docker — это виртуальная машина?
Нет. Виртуалка эмулирует целый компьютер с отдельной ОС — это тяжело. Контейнер делит ядро ОС хоста и изолирует только приложение. Поэтому он стартует за секунды и весит мегабайты, а не гигабайты. Одинаковость окружения даёт, а накладных расходов виртуалки — нет.
В чём разница между image и build в compose?
image — взять готовый образ из реестра (Docker Hub), например postgres:18-alpine. build — собрать свой образ из Dockerfile. Готовое (база, кэш) тянут через image, своё приложение — собирают через build.
Почему сборка иногда мгновенная, а иногда долгая?
Из-за кэша слоёв. Если поменялся только код — пересобираются лишь нижние слои (копирование кода и сборка), а установка зависимостей берётся из кэша. Если тронул package.json — пересобирается и установка, отсюда долго. Поиграй в блоке «Слои и кэш» выше — там это видно.
Если удалить контейнер, потеряю данные?
Данные внутри контейнера — да. Данные в томе — нет. Поэтому у базы есть volumes: pgdata:/var/lib/postgresql: контейнер можно сносить и пересоздавать, а том с данными остаётся. Пропадёт всё только если удалить сам том.
Что такое multi-stage build?
Это когда в одном Dockerfile несколько стадий сборки (FROM … AS имя), а в финальный образ попадает только нужное. Например: собрали приложение в «тяжёлой» стадии со всеми инструментами, а в прод-образ скопировали лишь готовый результат — образ получается маленьким и без лишнего. Построчный пример — в справочнике.
итог одной строкой

Образ — рецепт, контейнер — блюдо, слои — кэш сборки, том — где данные, сеть — как все находят друг друга. Эти пять картинок держат весь Docker.
