Шейдерный фон в шапке, а не во весь экран

Полноэкранный шейдер — самый дорогой способ показать, что сайт современный. Он забирает весь вьюпорт, спорит с текстом и первым лезет в бюджет первого экрана. Полоса под шапкой решает ту же задачу за долю стоимости. Ниже — что выбрать, как поставить и какие ограничения обязательны.

Так это выглядит

Полоса ниже — не картинка и не видео. Это работающий фрагментный шейдер на чистом WebGL, без единой библиотеки. Кнопки под ним меняют оттенок — так же, как это будет работать при переходе между разделами сайта.

Последняя строка важнее прочих. Когда оттенок привязан к разделу, движение перестаёт быть украшением и начинает нести информацию: человек видит, что перешёл в другую часть сайта, ещё до того, как прочитает заголовок.

Что выбрать

Для узкой полосы главный критерий — вес библиотеки, а не богатство эффектов. Второй — можно ли задать цвет параметром, иначе оттенок по разделам превратится в четыре копии кода.

ИнструментВесКогда брать
Paper Shadersвыбор 0 зависимостей Готовые meshGradient, warp, waves, grainGradient. Цвета — параметры. Не тянет three.js.
OGL ~10 КБ Свой шейдер с минимальным весом. Нужно писать GLSL руками.
ShaderGradient средний Есть визуальный конструктор: настроил мышкой — забрал код.
shadcn shaders копипаста 57 компонентов, попадают в проект как свой код без зависимости.
Vanta.js ~120 КБ Только если нужно сложное взаимодействие с курсором.
Granim.js малый Без WebGL, обычный canvas. Годится как запасной вариант.
про 120 килобайт

Авторы Vanta честно сравнивают этот вес с фоновым видео — и там он выигрывает. Но для полосы в 90 пикселей тянуть целый 3D-движок несоразмерно. Вес библиотеки должен быть пропорционален площади, которую она закрашивает.

Установка за пять минут

1Поставить пакет
npm i @paper-design/shaders-react
2Обернуть в компонент

Цвета приходят снаружи — из настроек раздела, а не из кода компонента.

// components/ShaderBar.tsx import { MeshGradient } from "@paper-design/shaders-react"; /** * Полоса с переливом для шапки и подвала. * @param colors — палитра раздела, 2–4 цвета * @param height — высота полосы в пикселях */ export function ShaderBar({ colors, height = 90 }: Props) { return ( <div style={{ position: "absolute", inset: 0 }} aria-hidden> <MeshGradient colors={colors} speed={0.12} /> </div> ); }

speed 0.12 — намеренно медленно. Это фон, а не заставка.

3Уложить слои

Родителю — position: relative и overflow: hidden. Содержимому шапки — position: relative, иначе оно окажется под подложкой.

Свой шейдер

Если нужен точный контроль над картинкой, пишется фрагментный шейдер. Перелив получается наложением нескольких волн разной частоты — дешёвая замена настоящему шуму Перлина.

// Сумма синусов даёт мягкие полосы, похожие на ткань precision mediump float; uniform float uTime; uniform vec2 uSize; uniform vec3 uTint; // оттенок раздела void main() { vec2 uv = gl_FragCoord.xy / uSize; // три волны с разным периодом и скоростью float w = sin(uv.x * 3.0 + uTime * 0.30) + sin(uv.x * 7.0 - uTime * 0.21 + uv.y * 4.0) + sin(uv.y * 5.0 + uTime * 0.17); float v = w / 6.0 + 0.5; // приводим к 0…1 v = pow(v, 1.6) * 0.62; // приглушаем: сверху текст gl_FragColor = vec4(uTint * v, 1.0); }
множитель 0.62

Это не украшение, а требование к контрасту. Он держит яркость подложки низкой, чтобы светлый текст шапки оставался читаемым в самой светлой фазе анимации, а не в среднем по кадрам.

Правила, без которых не работает

Сам шейдер несложен. Сложно сделать так, чтобы он не жёг батарею. Всё перечисленное — обязательное.

ПравилоЗачем
Рендер только когда видноIntersectionObserver на полосу. Подвал внизу страницы не должен крутиться, пока до него не долистали.
Пауза вне вкладкиvisibilitychange. Иначе свёрнутая вкладка продолжает считать кадры.
30 кадров в секундуМедленному переливу шестьдесят не нужно. Расход вдвое ниже, разницы не видно.
Плотность до 1.5На экранах с devicePixelRatio 3 закрашивать честно незачем — у фона нет резких краёв.
Малая амплитудаПроверять контраст в самой светлой фазе, а не на среднем кадре.

Пропуск кадров и остановка вне экрана:

const FRAME = 1000 / 30; // целимся в 30 fps let last = 0, visible = false, active = true; function loop(now) { if (visible && active && now - last >= FRAME) { last = now; draw(now / 1000); } requestAnimationFrame(loop); } // полоса ушла за экран — перестаём считать new IntersectionObserver( ([e]) => { visible = e.isIntersecting; }, { threshold: 0 } ).observe(canvas); // вкладка свернулась — тоже document.addEventListener("visibilitychange", () => { active = !document.hidden; });

Доступность и запасной вариант

Фон декоративен — aria-hidden="true" на канвасе, чтобы скринридер не пытался его озвучить.

reduced motion

При prefers-reduced-motion эффект не отключается целиком — показывается стоп-кадр. Картинка остаётся, движение уходит: страница не становится беднее для тех, кого от анимации укачивает.

const reduced = matchMedia("(prefers-reduced-motion: reduce)"); if (reduced.matches) { draw(0); // один кадр и остановка } else { requestAnimationFrame(loop); }

Поддержку WebGL надо проверять, а не предполагать: контекст может не создаться на старом устройстве или при программном рендеринге. Запасной вариант — обычный CSS-градиент того же оттенка. Пустой чёрной полосы пользователь видеть не должен.

const gl = canvas.getContext("webgl", { antialias: false }); if (!gl) { canvas.remove(); bar.style.background = `linear-gradient(100deg, ${tint}, transparent)`; }

Вопросы

Не будет ли это тормозить на мобильном?
При соблюдении правил выше — нет. Но на мобильном разумно вообще не запускать WebGL и показывать статичный градиент: экран маленький, полоса едва заметна, а батарея дороже. Ориентироваться на тип устройства, а не пытаться замерить производительность — в вебе это гадание.
Почему не GIF или видео?
Видео весит сотни килобайт против пары килобайт шейдера, не тянется под любую ширину без обрезки и не умеет менять оттенок параметром. Плюс автовоспроизведение видео на мобильных — отдельная история с ограничениями браузеров.
Как связать оттенок с разделом?
Передавать цвет параметром, а сам параметр брать из настроек раздела в CMS. Тогда новый раздел не потребует правки кода — только записи в админке. В демо выше это и происходит: кнопки меняют uTint, шейдер один.
А если дизайнер захочет эффект посложнее?
Сложность шейдера почти не влияет на вес — влияет на количество операций на пиксель. Для полосы в 90 пикселей запас большой. Опасно другое: яркие эффекты с высокой амплитудой, из-за которых текст поверх перестаёт читаться. Ограничение по контрасту важнее ограничения по сложности.
