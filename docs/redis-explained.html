Redis — быстрая память системы

PostgreSQL — надёжный склад, где данные лежат долго. Redis — блокнот под рукой: хранит в оперативной памяти, отвечает за микросекунды, но по своей природе временный. Его берут для кэша, сессий, счётчиков и очередей. Ниже — как он устроен, поиграй с ключами и посмотри, как они протухают по таймеру.

Что это простыми словами

Redis хранит данные как ключ → значение: по имени ключа мгновенно достаёшь то, что под ним лежит. Как большой Map в коде, только общий для всех процессов и живущий отдельным сервисом. Главная суперсила и главное ограничение — одно и то же: он держит всё в оперативной памяти.

⚡ почему быстрый
данные в RAM

Не нужно идти на диск — ответ приходит за микросекунды. Идеально для того, что читается очень часто.

⏳ почему временный
память не резиновая

RAM меньше и дороже диска. Поэтому в Redis кладут то, что не жалко потерять или что легко пересоздать — часто с TTL (сроком жизни).

Плейграунд: ключи и TTL

Нажимай команды и смотри на хранилище. Ключ с TTL (EX — expire) сам исчезнет, когда таймер дойдёт до нуля. GET покажет HIT (ключ есть) или MISS (нет или уже протух). Это и есть повседневная работа с Redis.

Замечаешь? cache:user:7 протухает первым (TTL 8), session:42 — следом (TTL 15), а config без TTL живёт, пока его не удалят. GET по протухшему ключу — MISS: приложение поймёт, что нужно сходить в базу и пересоздать кэш.

Redis vs PostgreSQL — не соперники, а пара

Их часто ставят вместе именно потому, что они про разное. Postgres — источник правды, Redis — быстрый слой перед ним.

правило большого пальца

Если данные нельзя потерять — это PostgreSQL. Если это ускорение или временное состояние, что можно пересоздать — это Redis.

Зачем Redis в веб-приложении

🚀 кэш

Тяжёлый запрос посчитали один раз — результат положили в Redis на N секунд. Следующие запросы берут готовое, не мучая базу.

🔑 сессии

«Кто сейчас залогинен» читается на каждом запросе — держать это в памяти быстрее, чем в базе. Ключ — по токену сессии.

⏱ rate-limit

Счётчик запросов на IP с TTL: «не больше 100 в минуту». Redis умеет атомарно increment и сам сбрасывает счётчик по истечении.

📮 очереди

Фоновые задачи (отправка письма, обработка файла) складывают в очередь на Redis — воркеры разбирают их отдельно. Так делает BullMQ.

📢 pub/sub

Один процесс публикует событие, другие — подписаны и получают его. Простой способ связать несколько инстансов приложения.

🔒 блокировки

Распределённый замок: «эту задачу уже кто-то делает». Гарантирует, что операция не выполнится дважды параллельно.

Redis в netroom прямо сейчас

Честно, без прикрас: контейнер Redis уже поднят и подключён к сети, а адрес прописан в .env. Использование в коде сервера (кэш, сессии, очереди) — следующий шаг по плану. Инфраструктура готова и ждёт.

netroom-server/docker-compose.yml · сервис redis
redis: image: redis:8-alpine
Готовый образ Redis 8 (alpine — минимальный). Ничего не собираем, просто тянем из Docker Hub.
ports: - "6379:6379"
Стандартный порт Redis. Локально проброшен на хост — удобно заглянуть через GUI. В проде порт не открывают — Redis остаётся в приватной сети.
volumes: - redisdata:/data
Именованный том. Redis умеет сбрасывать снимок памяти на диск в /data — том нужен, чтобы этот снимок пережил перезапуск контейнера.
netroom-server/.env · адрес для приложения
REDIS_URL=redis://redis:6379
redis:// — схема (протокол Redis). Второе redis — имя сервиса из compose (хост в приватной сети, не localhost). 6379 — порт. По этому адресу приложение подключится, когда мы добавим клиент в код.

«В памяти» — значит данные пропадут?

Резонный страх. Ответ: Redis умеет сохранять на диск, но это его подстраховка, а не главное назначение.

RDB · снимок

Периодически сбрасывает снимок всей памяти в файл. Быстро, но между снимками часть данных можно потерять при сбое.

AOF · журнал

Пишет каждую операцию в журнал. Надёжнее, восстановление точнее, но чуть медленнее и файл больше.

Том redisdata:/data как раз хранит эти файлы. Но философия остаётся: в Redis не кладут то, что обязано жить вечно — для этого есть PostgreSQL. Redis переживёт перезапуск, но источником правды его не делают.

Частые вопросы

Redis заменяет базу данных?
Нет. Он дополняет её. Главные данные — в PostgreSQL, а Redis ускоряет доступ к тому, что читается часто, и хранит временное состояние (сессии, счётчики). Убери Redis — приложение станет медленнее, но данные не потеряются. Убери Postgres — потеряешь всё.
Почему бы не хранить вообще всё в Redis, раз он быстрый?
Три причины: память дорогая и ограниченная (весь набор данных в RAM не влезет и будет дорого); нет сложных запросов (JOIN, фильтры, аналитика — это к SQL); меньше гарантий надёжности. Redis блистает на узкой задаче «быстро отдать по ключу», а не как универсальное хранилище.
Что такое TTL и EX?
TTL (time to live) — сколько ключу осталось жить. EX 15 при записи значит «удали через 15 секунд». Это идеально для кэша и сессий: данные сами убираются, не надо чистить вручную. В плейграунде выше это и видно — полоска у ключа убывает до нуля.
Почему в адресе redis://redis:6379 слово redis дважды?
Первое — схема (протокол), как https:// у сайтов. Второе — имя хоста, а хост тут — это имя сервиса redis из docker-compose. Внутри сети Docker сервисы находят друг друга по имени, поэтому не localhost, а redis. 6379 — порт.
Redis виден из интернета?
В проде — нет. Там у сервиса нет секции ports, он живёт в приватной сети Docker, и достучаться до него может только backend по имени redis. Открытый наружу Redis без пароля — классическая дыра, поэтому его прячут. Локально порт проброшен только для удобства разработки.
итог одной строкой

PostgreSQL — память надолго, Redis — память под рукой. Первый хранит правду, второй ускоряет к ней доступ и держит временное. Вместе — быстрый и надёжный бэкенд.
