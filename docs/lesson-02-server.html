Боевой сервер под HTTPS

Выкатываем четыре репы из Урока 01 на настоящий VPS. Наружу смотрит только Nginx (80/443), рядом certbot выпускает и автоматически продлевает сертификаты Let's Encrypt. Три поддомена — api, app, admin. Postgres и Redis публичных портов не имеют вообще.

~90 мин SSH · ufw · Docker · Nginx · certbot этап 02 плана нужен VPS + домен

Выбор железа с чего начать

Провайдер не важен — подойдёт любой VPS/облако. Важно, что именно брать под наш стек.

Виртуалка, а не выделенный сервер

VPS / облако — создаётся за минуты, почасовая оплата, легко пересоздать и увеличить. Выделенное железо — мощно и дорого, нужно под серьёзную нагрузку. Для старта и учёбы — однозначно виртуалка.

Общие или выделенные ядра

ВариантСутьКогда брать
shared vCPUядра делятся с соседяминаш случай: дёшево, для старта хватает
dedicated vCPUядра только твоистабильная прод-нагрузка, дороже
ARM64другая архитектурадешевле, но следи за совместимостью образов
про ARM — важный нюанс

ARM-тарифы дешевле, но это другая архитектура. Наши образы (node, postgres, redis, nginx, certbot) — мультиарх и под ARM работают. Но если попадётся образ только под x86 — на ARM он не запустится. Для предсказуемости на старте бери x86.

Сколько ресурсов нужно нашему стеку

На сервере крутятся: Nest, PostgreSQL, Redis, два Nginx со статикой, Nginx-прокси и certbot. Главный ограничитель — память.

  • RAM — сердце выбора. Postgres + Node + Redis: минимум 2 ГБ, комфортно 4 ГБ. На 2 ГБ жить можно, но тесно — особенно при сборке образов прямо на сервере.
  • vCPU — 2 ядра достаточно для старта.
  • Диск (SSD/NVMe) — 40 ГБ надолго хватит (образы + база). Следи за ростом базы.
рекомендация для старта

2 vCPU · 4 ГБ RAM · ~40 ГБ NVMe, x86, shared vCPU. Золотая середина — не тесно и недорого (несколько евро в месяц у любого провайдера). Совсем мало данных и трафика — хватит и 2 ГБ, но с запасом спокойнее.

Что ещё выбрать при создании

  • Локация — ближе к пользователям. Меньше пинг.
  • ОС — Ubuntu 24.04 LTS: именно на неё рассчитан этот урок.
  • SSH-ключ — добавь свой публичный ключ прямо в мастере (не пароль).
  • Бэкапы/снапшоты — если провайдер предлагает, для прода стоит включить; плюс свой pg_dump (в конце урока).
что нужно до старта

1. Пройден Урок 01 — четыре репы есть и локально всё поднимается. 2. Создан VPS (Ubuntu 24.04), есть его IP. 3. Есть реальный домен и доступ к его DNS. 4. Твой публичный SSH-ключ добавлен при создании. 5. Есть email для Let's Encrypt.

Что построим результат урока

Три приложения на трёх поддоменах за одним обратным прокси. Наружу открыты только 80/443 (Nginx). Приложения, база и кэш живут в приватной сети Docker.

браузер │ HTTPS (443) ▼ ┌─ VPS ─── ufw: наружу только 22 / 80 / 443 ──────────────────────────────┐ │ │ │ nginx (единственный публичный) certbot (продлевает TLS) │ │ ├── api.ДОМЕН ──► backend :3030 (NestJS) │ │ ├── app.ДОМЕН ──► client :80 (статика в nginx) │ │ └── admin.ДОМЕН ──► admin :80 (статика в nginx) │ │ │ │ │ backend ──► postgres · redis │ │ портов наружу НЕТ │ └────────────────────────────────────────────────────────────────────────┘
главное отличие от локалки — безопасность

Локально мы публиковали 5432 и 6379 для удобства. На публичном сервере это дыра: база с интернет-доступом взламывается ботами за часы. В compose.prod.yml у них вообще нет секции ports — backend достаёт их по приватной сети как postgres / redis.

Шаг 2 — Защита сервера безопасность

на сервере (под deploy)bash
# firewall: наружу только SSH, HTTP и HTTPS
sudo ufw allow OpenSSH
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable

# бан за перебор
sudo apt update
sudo apt install -y fail2ban

# автоматические обновления безопасности
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
Docker умеет обходить ufw — знай об этом

Docker правит iptables напрямую, и опубликованный порт может стать виден из интернета даже при включённом ufw. Поэтому главная защита базы — вообще не публиковать её порт, что мы и делаем. Firewall — второй рубеж, а не единственный.

Шаг 7 — compose.prod.yml ядро прода

Главный файл деплоя: собирает три приложения из соседних реп (стадия prod), поднимает приватные Postgres/Redis и пару nginx + certbot.

netroom-infra/compose.prod.ymlyaml · prod
services:
  backend:
    build:
      context: ../netroom-server
      target: prod
    environment:
      NODE_ENV: production
      PORT: 3030
      # Browser-facing origins allowed to call the API.
      CORS_ORIGIN: https://app.${DOMAIN},https://admin.${DOMAIN}
      # Private, server-to-server connections over the compose network.
      DATABASE_URL: postgres://netroom:${POSTGRES_PASSWORD}@postgres:5432/netroom
      REDIS_URL: redis://redis:6379
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started
    restart: unless-stopped

  client:
    build:
      context: ../netroom-client
      target: prod
      args:
        # Baked into the static bundle at build time.
        VITE_API_URL: https://api.${DOMAIN}
    restart: unless-stopped

  admin:
    build:
      context: ../netroom-admin
      target: prod
      args:
        VITE_API_URL: https://api.${DOMAIN}
    restart: unless-stopped

  postgres:
    image: postgres:18-alpine
    environment:
      POSTGRES_USER: netroom
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: netroom
    # No published ports — reachable only inside the private network.
    volumes:
      - pgdata:/var/lib/postgresql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U netroom"]
      interval: 5s
      timeout: 3s
      retries: 5
    restart: unless-stopped

  redis:
    image: redis:8-alpine
    # No published ports — reachable only inside the private network.
    volumes:
      - redisdata:/data
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    environment:
      DOMAIN: ${DOMAIN}
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/templates:/etc/nginx/templates:ro
      - ./nginx/snippets:/etc/nginx/snippets:ro
      - certbot_conf:/etc/letsencrypt:ro       # certificates (written by certbot)
      - certbot_www:/var/www/certbot:ro        # ACME challenge webroot
    depends_on:
      - backend
      - client
      - admin
    # Render the config (substitute only ${DOMAIN}, keep nginx's own $vars — we do
    # envsubst by hand because overriding the command skips the image's auto-render),
    # then start nginx and reload every 6h so renewed certificates are picked up.
    command: >
      /bin/sh -c "envsubst '$${DOMAIN}' < /etc/nginx/templates/default.conf.template > /etc/nginx/conf.d/default.conf;
      while :; do sleep 6h & wait $${!}; nginx -s reload; done & exec nginx -g 'daemon off;'"
    restart: unless-stopped

  certbot:
    image: certbot/certbot
    volumes:
      - certbot_conf:/etc/letsencrypt
      - certbot_www:/var/www/certbot
    # Attempt renewal twice a day; certbot only acts when a cert is near expiry.
    entrypoint: >
      /bin/sh -c "trap exit TERM; while :; do certbot renew --webroot -w /var/www/certbot; sleep 12h & wait $${!}; done"
    restart: unless-stopped

volumes:
  pgdata:
  redisdata:
  certbot_conf:
  certbot_www:
что делает этот файл прод-конфигом

1. target: prod — боевая стадия multi-stage Dockerfile (собранный dist под non-root; фронты — статика в nginx). 2. restart: unless-stopped у всех — переживают перезагрузку и падения. 3. У postgres/redis нет ports. 4. Публикует 80/443 только nginx. 5. certbot крутится рядом и продлевает сертификаты.

две хитрости в этих командах

nginx: envsubst '$${DOMAIN}' подставляет только ${DOMAIN} и не трогает собственные переменные nginx ($host, $scheme) — иначе конфиг сломается. Цикл sleep 6h; nginx -s reload нужен, чтобы подхватить обновлённый сертификат без перезапуска. certbot: цикл renew раз в 12 часов — сам certbot ничего не делает, пока сертификат не близок к истечению.

Шаг 8 — Конфиг Nginx маршруты + TLS

Шаблон рендерится при старте контейнера. Один блок :80 — для ACME-проверки и редиректа на HTTPS; три блока :443 — по одному на поддомен.

🔗 узнать больше

Что такое reverse-proxy, как Nginx роутит по поддоменам и зачем TLS-терминация — с интерактивом и построчным разбором этого же конфига в Nginx: reverse-proxy.

netroom-infra/nginx/templates/default.conf.templatenginx
# Maps the Upgrade header to the Connection value: "upgrade" for WebSockets,
# "close" otherwise. Referenced by snippets/proxy.conf.
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

# ---- HTTP (:80): ACME challenge + redirect to HTTPS ----
server {
    listen 80;
    listen [::]:80;
    server_name api.${DOMAIN} app.${DOMAIN} admin.${DOMAIN};

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;          # certbot writes challenge files here
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# ---- HTTPS: api.${DOMAIN} -> backend ----
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;
    server_name api.${DOMAIN};

    ssl_certificate     /etc/letsencrypt/live/api.${DOMAIN}/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.${DOMAIN}/privkey.pem;
    include /etc/nginx/snippets/ssl-params.conf;

    location / {
        include /etc/nginx/snippets/proxy.conf;
        proxy_pass http://backend:3030;
    }
}

# ---- HTTPS: app.${DOMAIN} -> client ----
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;
    server_name app.${DOMAIN};

    ssl_certificate     /etc/letsencrypt/live/api.${DOMAIN}/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.${DOMAIN}/privkey.pem;
    include /etc/nginx/snippets/ssl-params.conf;

    location / {
        include /etc/nginx/snippets/proxy.conf;
        proxy_pass http://client:80;
    }
}

# ---- HTTPS: admin.${DOMAIN} -> admin ----
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;
    server_name admin.${DOMAIN};

    ssl_certificate     /etc/letsencrypt/live/api.${DOMAIN}/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.${DOMAIN}/privkey.pem;
    include /etc/nginx/snippets/ssl-params.conf;

    location / {
        include /etc/nginx/snippets/proxy.conf;
        proxy_pass http://admin:80;
    }
}
почему все три блока смотрят в live/api.ДОМЕН

certbot выпускает один сертификат сразу на три поддомена (SAN). Он лежит в папке по имени первого домена — api.ДОМЕН. Поэтому все server-блоки ссылаются на один и тот же файл. Это нормально и экономит выпуски.

Общие куски вынесены в snippets — чтобы не дублировать в каждом блоке:

netroom-infra/nginx/snippets/proxy.confnginx
# Common reverse-proxy headers. Included inside every proxied location.
# Passes the real client identity upstream and is ready for WebSocket upgrades.

proxy_http_version 1.1;
proxy_set_header Host              $host;
proxy_set_header X-Real-IP         $remote_addr;
proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
proxy_set_header Upgrade           $http_upgrade;
proxy_set_header Connection        $connection_upgrade;
netroom-infra/nginx/snippets/ssl-params.confnginx
# TLS hardening — Mozilla "Intermediate" profile (2024/2025).
# Included by every HTTPS server block. Certificate paths stay in the server
# block itself (they are domain-specific); everything reusable lives here.

ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers off;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;

ssl_session_timeout 1d;
ssl_session_cache shared:SSL:10m;
ssl_session_tickets off;

# No ssl_dhparam: only ECDHE ciphers are enabled, so DH parameters are unused.
# No OCSP stapling: Let's Encrypt is retiring OCSP in 2025.

# Security headers (sent on HTTPS responses).
add_header Strict-Transport-Security "max-age=63072000" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "SAMEORIGIN" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;

Шаг 9 — Первый выпуск сертификата курица и яйцо

Тут есть классическая ловушка: nginx не стартует без сертификата (в конфиге прописан путь к нему), а сертификат не выпустить без работающего nginx (Let's Encrypt должен забрать ACME-проверку по HTTP). Скрипт init-letsencrypt.sh разрывает этот круг.

netroom-infra/init-letsencrypt.shsh · запускается один раз
#!/bin/sh
# One-time TLS bootstrap for the netroom stack (Nginx + certbot / Let's Encrypt).
#
# Solves the chicken-and-egg problem: nginx won't start without a certificate,
# but the certificate can't be issued without nginx answering the ACME challenge.
# Steps: dummy self-signed cert -> start nginx -> replace with a real cert -> reload.
#
# Run ONCE on the server, after `cp .env.example .env` (set DOMAIN + CERTBOT_EMAIL)
# and after DNS A-records api/app/admin point to this host:
#   chmod +x init-letsencrypt.sh && ./init-letsencrypt.sh
#
# Tip: set STAGING=1 the first time to use Let's Encrypt's staging CA and avoid
# hitting rate limits while you debug, then re-run without it for a trusted cert.

Что он делает по шагам:

  1. Фиктивный сертификат — самоподписанный, чтобы nginx смог стартовать.
  2. Поднимает nginx — теперь он отвечает на /.well-known/acme-challenge/.
  3. Удаляет фиктивный — освобождает путь под настоящий.
  4. Запрашивает настоящий у Let's Encrypt (webroot) сразу на три поддомена.
  5. Перезагружает nginx — уже с боевым сертификатом.
на сервере · netroom-infra/bash
# первый прогон — лучше на тестовом CA, чтобы не спалить лимиты
chmod +x init-letsencrypt.sh
STAGING=1 ./init-letsencrypt.sh

# убедился, что всё проходит — теперь по-настоящему
./init-letsencrypt.sh
про лимиты Let's Encrypt — не игнорируй STAGING

У Let's Encrypt жёсткие лимиты на количество выпусков для домена в неделю. Если отлаживаешь и повторяешь выпуск — легко упереться и остаться без сертификата на несколько дней. STAGING=1 использует тестовый центр (сертификат будет «недоверенным», но весь путь проверяется). Убедился, что работает — прогоняй без него.

этот деплой — ещё не финал

Сейчас образы собираются на сервере: это ест его CPU/RAM и требует исходников на хосте. Профессиональный следующий шаг — собирать образы в CI (GitHub Actions), пушить в реестр (GHCR) и на сервере делать только pull. Это этап 03: после коммита в main деплой едет сам.

🔗 а можно проще — через панель

Всё это (Nginx, certbot, поддомены, база, деплой) умеет настраивать за тебя Coolify — self-hosted PaaS. Ручной путь этого урока — чтобы понимать, что происходит; Coolify — чтобы быстро катить. Один раз пройди руками, потом решай сам.

Частые вопросы

Почему поддомены, а не один домен с путями /api, /admin?

Потому что приложения независимы: свои репы, свои деплои, свои жизненные циклы. Поддомены отражают это честно — каждое живёт по своему адресу.

Плата — cross-origin, то есть CORS. Он и так нужен локально (разные порты), так что механизм один и тот же в dev и в prod.

Зачем нужен init-letsencrypt.sh, разве certbot не сам всё делает?

Проблема курицы и яйца: в конфиге nginx прописаны пути к сертификату — без файла он не стартует. А certbot не может выпустить сертификат, пока nginx не отвечает на ACME-проверку по HTTP.

Скрипт разрывает круг: кладёт самоподписанную «пустышку» → поднимает nginx → удаляет пустышку → запрашивает настоящий → перезагружает nginx. Дальше сервис certbot продлевает сам, скрипт больше не нужен.

Почему envsubst с кавычками — '$${DOMAIN}'?

Так envsubst подставляет только ${DOMAIN}. Без этого он съел бы и собственные переменные nginx ($host, $scheme, $remote_addr) — конфиг сломался бы.

А $$ нужно, чтобы Compose не подставил переменную сам: так до шелла в контейнере доедет обычный $.

Зачем nginx перезагружается каждые 6 часов?

Когда certbot продлевает сертификат, файл на диске меняется — но nginx держит старый в памяти. nginx -s reload заставляет его перечитать файлы, без разрыва соединений.

Раз в 6 часов — с запасом: сертификат живёт 90 дней, certbot обновляет заранее.

Почему VITE_API_URL передаётся через args, а не environment?

Vite запекает VITE_-переменные в бандл при сборке. В браузер уезжает готовый JS — рантайм-переменную ему подсунуть некуда.

Поэтому адрес API идёт в сборку образа (build.args). У backend наоборот — environment, потому что Node читает env в рантайме.

Почему у postgres и redis совсем нет ports?

Снаружи к ним никто обращаться не должен. Backend достаёт их по приватной сети Docker — по именам postgres и redis. Публиковать порт = открыть базу интернету.

Нужно подключиться psql'ем к прод-базе? Через SSH-туннель, а не открывая порт миру.

Почему dev и prod — разные файлы, а не оверрайды?

compose.dev.yml просто включает compose'ы приложений (порты наружу, hot-reload), а compose.prod.yml — самостоятельный боевой файл (nginx, certbot, закрытые БД, build-args). Их не запускают вместе.

Общее у dev и prod — Dockerfile каждого приложения: одна multi-stage сборка, просто разные стадии.

что у тебя теперь есть

Боевой стек на одном сервере: три приложения на HTTPS-поддоменах за Nginx, сертификаты выпускаются и продлеваются сами, база и кэш закрыты от интернета, TLS настроен по Mozilla Intermediate, автозапуск после перезагрузки, бэкапы. Осталось автоматизировать выкат — этап 03 (CI/CD).
