урок 1.2 · боевой сервер

Боевой сервер на Hetzner

Берём стек из Урока 1.1 и поднимаем его на настоящем сервере: доступ по ключу, защита, Docker, безопасный прод-compose (база и кэш закрыты от интернета), Nginx и HTTPS. К концу — сервис открывается по https://твой-домен.

~90 мин SSH · ufw · Docker · Nginx этап 01 · часть 2 из 2 нужен VPS (Hetzner) + домен
что нужно до старта

1. Пройден Урок 1.1 — стек крутится локально. 2. Заведён VPS на Hetzner (Ubuntu 24.04), есть его IP. 3. Есть домен (или поддомен) и доступ к его DNS. 4. Твой публичный SSH-ключ добавлен в сервер при создании (Hetzner предлагает это в мастере).

Что построим результат урока

Тот же стек, но по-боевому. Наружу смотрит только Nginx (порты 80/443). Он проксирует на приложение, а Postgres и Redis не публикуются вообще — живут во внутренней сети Docker и доступны только приложению.

интернет │ https://api.твой-домен (443) ▼ ┌─ сервер (Hetzner) ─── ufw: наружу только 22 / 80 / 443 ──────────┐ │ │ │ Nginx ──proxy──► app :3030 ──► postgres / redis │ │ 443/TLS 127.0.0.1 без публикации портов! │ └─────────────────────────────────────────────────────────────────┘
главное отличие от локалки — безопасность

Локально мы публиковали порты 5432 и 6379 для удобства. На публичном сервере это дыра: база с интернет-доступом и слабым паролем взламывается ботами за часы. В прод-конфиге эти порты не публикуем — приложение всё равно видит их по внутренней сети (postgres:5432). Разберём это в шаге 5.

Шаг 1 — Доступ по SSH безопасность

Первый вход — под root по ключу. Сразу заводим отдельного пользователя (работать под root — плохая привычка) и переносим ему наш ключ.

на своей машинеbash
# первый вход на сервер (IP выдал Hetzner)
ssh root@ТВОЙ_IP
на сервереbash
# создать пользователя deploy и дать ему sudo
adduser deploy
usermod -aG sudo deploy

# перенести наш SSH-ключ пользователю deploy
rsync --archive --chown=deploy:deploy ~/.ssh /home/deploy

Проверь, что заходишь под новым пользователем (в новом окне терминала, не закрывая root):

на своей машинеbash
ssh deploy@ТВОЙ_IP

Работает? Тогда закрываем вход под root и по паролю. На сервере отредактируй SSH-конфиг:

на сервере (под deploy)bash
sudo nano /etc/ssh/sshd_config
sshd_config — найти и выставитьconfig
PermitRootLogin no
PasswordAuthentication no
на сервереbash
# применить изменения
sudo systemctl restart ssh
не закрывай текущую сессию, пока не проверил

После рестарта SSH открой новое окно и убедись, что вход под deploy по ключу работает. Только потом закрывай старую сессию. Иначе рискуешь запереть себя снаружи.

Шаг 2 — Защита сервера безопасность

Три вещи базовой гигиены: firewall (пускать только нужные порты), fail2ban (банить перебор) и авто-обновления безопасности.

на сервере (под deploy)bash
# firewall: разрешить только SSH, HTTP и HTTPS
sudo ufw allow OpenSSH
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable

# бан за перебор паролей/ключей
sudo apt update
sudo apt install -y fail2ban

# автоматические обновления безопасности
sudo apt install -y unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades
почему в firewall нет 5432 и 6379

Именно поэтому. Наружу открыты только 22/80/443. Порты базы и кэша не открываем — и в прод-compose их тоже не публикуем (шаг 5). Два рубежа вместо одного.

Docker умеет обходить ufw — знай об этом

Docker правит iptables напрямую и публикация порта в compose может стать видна из интернета даже при включённом ufw. Поэтому нельзя полагаться только на firewall: главная защита базы — вообще не публиковать её порт в прод-compose.

Шаг 3 — Docker на сервере инфра

Ставим Docker Engine и compose-плагин официальным скриптом, добавляем себя в группу docker.

на сервереbash
# официальный установщик Docker
curl -fsSL https://get.docker.com | sh

# запускать docker без sudo (перелогинься после этого)
sudo usermod -aG docker deploy

# проверка
docker --version
docker compose version
чек-поинт

Заходишь на сервер по ключу под deploy, root и пароль закрыты, ufw включён (22/80/443), docker compose version отвечает. Фундамент готов.

Шаг 4 — Код на сервер деплой

Забираем репозиторий dev-vault на сервер через git. Пока — вручную; на этапе 03 это будет делать GitHub Actions автоматически после коммита.

на сервере (под deploy)bash
# склонировать проект (для приватной репы понадобится deploy-ключ)
git clone https://github.com/GrafSoul/dev-vault.git
cd dev-vault

Шаг 5 — Безопасный прод-compose безопасность

Ключевой шаг. Заводим отдельный docker-compose.prod.yml — он отличается от локального тремя вещами: боевой образ (prod-стадия, не dev), restart-политики (переживать перезагрузку и падения) и никаких публичных портов у базы и кэша.

docker-compose.prod.ymlyaml
services:
  app:
    build:
      context: .
      target: prod            # боевой образ: node dist/main.js (не dev)
    restart: unless-stopped   # сам поднимется после перезагрузки/падения
    env_file: .env
    ports:
      - "127.0.0.1:3030:3030" # виден ТОЛЬКО серверу (для Nginx), не из интернета
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started

  postgres:
    image: postgres:17-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: devvault
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}   # из .env, сильный пароль
      POSTGRES_DB: devvault
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devvault"]
      interval: 5s
      timeout: 3s
      retries: 5
    # ← НЕТ секции ports: база не смотрит в интернет

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    volumes:
      - redisdata:/data
    # ← НЕТ секции ports: Redis только во внутренней сети

volumes:
  pgdata:
  redisdata:
сравни с локальным compose

Три отличия: target: prod (собранный dist вместо dev-watch), restart: unless-stopped у всех, и у postgres/redis убрана секция ports. У app порт привязан к 127.0.0.1 — снаружи не достучаться, только Nginx на самом сервере. Приложение по-прежнему ходит к базе по postgres:5432 внутри сети Docker.

Шаг 6 — Прод-.env с сильными паролями безопасность

На сервере создаём .env из шаблона и вписываем настоящие пароли — не devvault из локалки. Сгенерируй случайный:

на сервере · dev-vault/bash
# сгенерировать сильный пароль
openssl rand -base64 24

# создать рабочий .env из шаблона и отредактировать
cp .env.example .env
nano .env
.env (на сервере)env
NODE_ENV=production
PORT=3030

# сильный пароль, сгенерированный openssl (одинаковый в обеих строках)
POSTGRES_PASSWORD=ВСТАВЬ_СГЕНЕРИРОВАННЫЙ_ПАРОЛЬ
DATABASE_URL=postgresql://devvault:ВСТАВЬ_СГЕНЕРИРОВАННЫЙ_ПАРОЛЬ@postgres:5432/devvault
REDIS_URL=redis://redis:6379
почему пароль в двух местах

POSTGRES_PASSWORD задаёт пароль базе, а DATABASE_URL — как приложение к ней подключается. Значения должны совпадать. .env в .gitignore, поэтому реальные пароли в git не попадают — только шаблон.

Шаг 7 — Запуск на сервере деплой

Поднимаем стек прод-конфигом в фоне (-d):

на сервере · dev-vault/bash
# собрать боевой образ и поднять стек в фоне
docker compose -f docker-compose.prod.yml up -d --build

# проверить статусы
docker compose -f docker-compose.prod.yml ps

# проверить, что приложение отвечает локально на сервере
curl http://127.0.0.1:3030

curl вернул Hello World! — приложение живо, но пока доступно только внутри сервера. Осталось открыть его миру через Nginx.

Шаг 8 — Nginx и HTTPS инфра

Nginx примет трафик из интернета на 80/443 и проксирует на приложение. Сертификат HTTPS бесплатно выдаст Let's Encrypt через certbot. Сначала — направь домен на сервер: в DNS добавь A-запись api.твой-домен → IP сервера.

на сервереbash
sudo apt install -y nginx
sudo nano /etc/nginx/sites-available/dev-vault
/etc/nginx/sites-available/dev-vaultnginx
server {
    server_name api.твой-домен;

    location / {
        proxy_pass http://127.0.0.1:3030;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
на сервереbash
# включить сайт и проверить конфиг
sudo ln -s /etc/nginx/sites-available/dev-vault /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

# выпустить HTTPS-сертификат (certbot сам пропишет 443 и редирект)
sudo apt install -y certbot python3-certbot-nginx
sudo certbot --nginx -d api.твой-домен
проверь себя — сервис в бою

Открой https://api.твой-домен в браузере — видишь Hello World! по HTTPS. База и Redis наружу закрыты, стек переживает sudo reboot (проверь!). Этап 01 полностью выполнен — и локально, и на сервере.

Обновление и полезные команды

на сервере · dev-vault/bash
# выкатить новую версию после изменений в репе
git pull
docker compose -f docker-compose.prod.yml up -d --build

# логи приложения
docker compose -f docker-compose.prod.yml logs -f app

# перезапустить стек
docker compose -f docker-compose.prod.yml restart

# бэкап базы в файл
docker compose -f docker-compose.prod.yml exec postgres \
  pg_dump -U devvault devvault > backup_$(date +%F).sql
этот ручной деплой — временный

Сейчас обновление = git pull + up -d --build руками. На этапе 03 (CI/CD) это возьмёт на себя GitHub Actions: после коммита в main он сам соберёт образ и выполнит деплой на сервер. Ты понял верно — «после коммита автоматом заливает». Но сначала полезно один раз сделать руками, чтобы понимать, что автоматизируем.

Частые вопросы

Почему нельзя оставить порты Postgres/Redis открытыми, как локально?

Локально localhost доступен только тебе. На сервере публикация порта открывает базу всему интернету. Боты постоянно сканируют 5432/6379 и подбирают пароли — открытая база со слабым паролем живёт часы.

Решение простое: приложение ходит к базе по внутренней сети Docker (postgres:5432), а наружу порт не публикуется. Никто извне до базы не дотянется.

Зачем отдельный docker-compose.prod.yml, а не тот же файл?

У прода другие требования: боевой образ вместо dev-watch, restart-политики, закрытые порты БД, сильные пароли. Держать это отдельным файлом — честно и наглядно: видно, чем прод отличается от локалки.

Позже, когда почувствуешь Compose увереннее, общие части можно вынести и не дублировать (через override-файлы). Пока нагляднее — отдельным файлом.

А если у меня нет домена?

Для теста можно обращаться по http://IP_сервера (без HTTPS) — но только временно: без домена не выпустить нормальный сертификат, а гонять прод по HTTP небезопасно.

Домены стоят копейки. Для боевого сервиса он нужен — и ради HTTPS, и ради адекватного адреса. Certbot без домена работать не будет.

Что делает restart: unless-stopped?

Docker сам перезапустит контейнер, если он упал или если сервер перезагрузился — но не будет поднимать его, если ты остановил вручную (docker compose stop).

Это базовая надёжность: после reboot сервера сервис поднимется сам, без твоего участия.

Где физически данные и переживут ли они пересоздание контейнера?

В томах Docker (pgdata, redisdata) на диске сервера — как и локально. Пересборка и перезапуск контейнеров данные не трогают.

Но том живёт на одном диске сервера, поэтому делай бэкапы (команда pg_dump выше) и храни их отдельно — на случай смерти сервера.

Nginx лучше пакетом на хосте или контейнером?

Оба варианта рабочие. Здесь мы поставили Nginx пакетом на хост — так проще всего дружит с certbot (он сам правит конфиг и продлевает сертификат).

Контейнерный Nginx тоже возможен и удобен, когда фронт и бэк оборачиваешь в один стек — но настройка HTTPS там чуть сложнее. Для старта хостовый Nginx — меньше боли.

что у тебя теперь есть

Полный боевой комплект: стек dev-vault + Postgres + Redis работает локально (урок 1.1) и на сервере по HTTPS, с закрытой от интернета базой, автозапуском и бэкапами. Осталось автоматизировать выкат — это этап 03 (CI/CD).
