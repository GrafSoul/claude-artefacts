Локальная среда в Docker

Заворачиваем пустой Nest-проект dev-vault в контейнер и поднимаем рядом PostgreSQL и Redis — всё одной командой, с автоперезагрузкой при правке кода. К концу урока у тебя воспроизводимое окружение, одинаковое у тебя на машине и потом на сервере.

~40 мин Docker · Compose шаги плана 1.1–1.3 нужен установленный Docker Desktop

Что построим результат этапа

Три контейнера в одной изолированной сети Docker. Твоё приложение видит базу и кэш по именам сервисов (postgres, redis), а ты с машины — по localhost на проброшенных портах.

важно про этот урок

Здесь мы не подключаем Nest к базе в коде — это этап 05 (Prisma). Задача сейчас: чтобы всё крутилось рядом, было доступно и перезапускалось само. Инфраструктура вперёд, логика потом.

Мини-теория: 4 слова Docker

Только то, без чего дальше будет непонятно. Аналогии намеренно бытовые.

📦

образ (image)

Замороженный слепок: код + среда + зависимости. Как установочный диск — сам не работает, но с него запускают.

▶️

контейнер (container)

Запущенный образ — живой процесс. Из одного образа можно поднять хоть десять контейнеров.

💾

том (volume)

Место, где данные переживают перезапуск контейнера. Без него база потеряет всё при down.

🕸️

сеть (network)

Compose соединяет контейнеры в одну сеть, где они находят друг друга по имени сервиса, а не по IP.

Шаг 1 — Dockerfile план 1.1

Multi-stage: один файл описывает и лёгкий боевой образ, и стадию для разработки. Сейчас нам нужна стадия dev, но prod пусть будет сразу — пригодится на этапе 03.

Dockerfiledockerfile
# ---- база: общая для всех стадий ----
FROM node:22-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- dev: с dev-зависимостями и hot-reload ----
FROM base AS dev
RUN npm ci
COPY . .
CMD ["npm", "run", "start:dev"]

# ---- build: собираем прод-сборку ----
FROM base AS build
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --omit=dev

# ---- prod: минимальный образ для сервера ----
FROM node:22-alpine AS prod
WORKDIR /app
ENV NODE_ENV=production
COPY --from=build /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
COPY --from=build /app/package*.json ./
EXPOSE 3000
CMD ["node", "dist/main.js"]

почему multi-stage

Стадия dev монтирует твой код и следит за изменениями. Стадия prod не содержит ни dev-зависимостей, ни исходников — только собранный dist. Финальный образ выходит в разы меньше и безопаснее.

Шаг 2 — docker-compose.yml план 1.2

Описываем весь стек в одном файле: приложение + база + кэш, их тома, порты и связи. depends_on с проверкой здоровья гарантирует, что app стартует, только когда Postgres реально готов принимать соединения.

docker-compose.ymlyaml
services:
  app:
    build:
      context: .
      target: dev            # берём dev-стадию из Dockerfile
    command: npm run start:dev
    ports:
      - "3000:3000"
    env_file: .env
    volumes:
      - .:/app               # код с хоста внутрь — для hot-reload
      - /app/node_modules    # но node_modules берём из образа
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started

  postgres:
    image: postgres:17-alpine
    environment:
      POSTGRES_USER: devvault
      POSTGRES_PASSWORD: devvault
      POSTGRES_DB: devvault
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devvault"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redisdata:/data

volumes:
  pgdata:
  redisdata:

частая ловушка

Из контейнера app база доступна как postgres:5432, а не localhost:5432. localhost внутри контейнера — это сам контейнер. Имя сервиса из compose = сетевой адрес соседа.

про том node_modules

Строка /app/node_modules без хостового пути — это анонимный том. Он «прикрывает» node_modules из образа, чтобы твоя папка с хоста их не затёрла. Без этого приёма контейнер часто падает с «module not found».

Шаг 3 — запуск и hot-reload план 1.3

Из папки dev-vault — одна команда, которая соберёт образ и поднимет все три контейнера:

терминал · dev-vault/bash
# собрать образы и поднять весь стек (логи в этом же окне)
docker compose up --build

Когда в логах появится строка Nest «Nest application successfully started» — открывай:

браузерurl
http://localhost:3000

Увидишь Hello World! — приложение из контейнера отвечает наружу. Теперь проверим hot-reload:

  1. Открой src/app.service.ts и поменяй строку в методе getHello() (там лежит 'Hello World!').
  2. Сохрани файл — в логах контейнера Nest сам пересоберётся за секунду.
  3. Обнови вкладку — текст изменился, хотя ты ничего не перезапускал руками.
проверь себя

Три контейнера в docker compose ps со статусом up, приложение отвечает на localhost:3000, правка кода подхватывается без ручного перезапуска. Если так — этап 01 выполнен.

Полезные команды, пока стек работает

терминалbash
# какие контейнеры запущены и их статус
docker compose ps

# живые логи только приложения
docker compose logs -f app

# зайти внутрь контейнера app (оболочка)
docker compose exec app sh

# подключиться к базе прямо в контейнере
docker compose exec postgres psql -U devvault -d devvault

# остановить стек (данные в томах сохранятся)
docker compose down

# остановить И удалить тома (полный сброс данных базы)
docker compose down -v

Шпаргалка команд Docker

Базовый набор на каждый день. compose — для всего стека, «голый» docker — для отдельных контейнеров и уборки.

docker compose up -dподнять стек в фоне (без логов в окне)
docker compose up --buildпересобрать образы и поднять
docker compose downостановить и удалить контейнеры
docker compose restart appперезапустить один сервис
docker compose logs -f appследить за логами сервиса
docker compose exec app shоболочка внутри контейнера

что у тебя теперь есть

Воспроизводимое окружение: dev-vault + Postgres + Redis поднимаются одной командой, код перезагружается на лету, данные базы живут в томе. Ровно этот же docker-compose.yml станет основой боевого стека на сервере (этап 02).
