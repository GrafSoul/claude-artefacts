Локальная среда в Docker

Собираем весь стек: сервер (Nest), клиент (React), PostgreSQL и Redis — и поднимаем его одной командой из корня. Две репы держат свои Docker-файлы, а тонкий корневой compose через include запускает всё вместе.

~50 мин Docker · Compose · include этап 01 плана нужен установленный Docker Desktop

Что построим результат этапа

Четыре контейнера в одной сети Docker. Так как корневой compose через include собирает обе репы в один проект, все сервисы попадают в общую сеть и видят друг друга по именам. Наружу порты проброшены на твою машину.

важно про этот урок

Код не подключаем к базе (это этап 05, Prisma) и не пишем логику. Задача — чтобы весь стек поднимался одной командой, был доступен и перезапускался сам. Инфраструктура вперёд, логика потом.

Структура: две репы + корневой оркестратор

Сервер и клиент — отдельные репозитории, каждый самодостаточный (свои Dockerfile и docker-compose.yml). Рядом — тонкая рабочая папка dev-vault/ с корневым compose, который ничего не дублирует, а просто через include подключает обе репы. Запуск — из неё.

# dev-vault/ — рабочая папка, запускаем отсюда ├── docker-compose.yml # корневой: include обеих реп ├── dev-vault-server/ # репа бэка (Nest) └── dev-vault-client/ # репа фронта (React)
почему так, а не один общий проект

Репы независимы: свои коммиты, свой деплой, можно вести разными людьми. Корневой compose — лишь удобство для локального запуска всего разом. На сервере каждый сервис едет сам по себе (этап 02).

Мини-теория: 4 слова Docker

Только то, без чего дальше будет непонятно. Аналогии намеренно бытовые.

📦

образ (image)

Замороженный слепок: код + среда + зависимости. Как установочный диск — сам не работает, но с него запускают.

▶️

контейнер (container)

Запущенный образ — живой процесс. Из одного образа можно поднять хоть десять контейнеров.

💾

том (volume)

Место, где данные переживают перезапуск контейнера. Без него база потеряет всё при down.

🕸️

сеть (network)

Compose соединяет контейнеры в одну сеть, где они находят друг друга по имени сервиса, а не по IP.

Сервер — dev-vault-server backend

Внутри репы dev-vault-server уже есть Nest (nest new). Добавляем 5 файлов:

# dev-vault-server/ ├── src/ # уже есть (nest new) ├── package.json # уже есть ├── Dockerfile # ← добавляем ├── .dockerignore # ← добавляем ├── docker-compose.yml # ← добавляем ├── .env.example # ← шаблон, коммитится └── .env # ← cp .env.example .env (в .gitignore)

Dockerfile

Multi-stage: общая база → deps (ставит зависимости один раз) → dev и build её переиспользуют. Прод-образ — минимальный и под non-root (USER node).

dev-vault-server/Dockerfiledockerfile
# syntax=docker/dockerfile:1

# ---- base: shared by all stages ----
FROM node:24-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- deps: install dependencies (cacheable layer) ----
FROM base AS deps
RUN npm ci

# ---- dev: dev dependencies + hot-reload ----
FROM deps AS dev
COPY . .
CMD ["npm", "run", "start:dev"]

# ---- build: production build ----
FROM deps AS build
COPY . .
RUN npm run build
RUN npm prune --omit=dev

# ---- prod: minimal image for the server ----
FROM node:24-alpine AS prod
WORKDIR /app
ENV NODE_ENV=production

COPY --from=build --chown=node:node /app/node_modules ./node_modules
COPY --from=build --chown=node:node /app/dist ./dist
COPY --from=build --chown=node:node /app/package*.json ./

USER node
EXPOSE 3030
CMD ["node", "dist/main.js"]

docker-compose.yml

Три сервиса репы: app (Nest), postgres, redis. depends_on с проверкой здоровья держит старт app, пока база реально не готова.

Postgres и Redis ставить НЕ надо

Строки image: postgres:18-alpine и image: redis:8-alpine — команда Docker «скачай готовый образ». При первом up он подтянет их сам. На машине нужен только Docker.

dev-vault-server/docker-compose.ymlyaml
services:
  app:
    build:
      context: .
      target: dev            # use the dev stage from the Dockerfile
    command: npm run start:dev
    ports:
      - "3030:3030"
    env_file: .env
    volumes:
      - .:/app               # mount host code into the container for hot-reload
      - /app/node_modules    # but keep node_modules from the image
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started

  postgres:
    image: postgres:18-alpine
    environment:
      POSTGRES_USER: devvault
      POSTGRES_PASSWORD: devvault
      POSTGRES_DB: devvault
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devvault"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:8-alpine
    ports:
      - "6379:6379"
    volumes:
      - redisdata:/data

volumes:
  pgdata:
  redisdata:
две ловушки, которые легко словить

1. В postgres:18 путь данных — /var/lib/postgresql (без /data, как было в 17). 2. Из контейнера app база доступна как postgres:5432, а не localhost — localhost внутри контейнера это сам контейнер, а имя сервиса из compose = адрес соседа.

.env.example и рабочий .env

Шаблон .env.example коммитим, рабочий .env — нет (в нём потом реальные пароли). Обрати внимание на CORS_ORIGIN — адрес фронта, которому разрешено звать API (про это ниже).

dev-vault-server/.env.exampleenv
# ---- Application ----
# Port must match the port in docker-compose.yml and EXPOSE in the Dockerfile
PORT=3030
NODE_ENV=development

# Frontend origin allowed to call the API (CORS)
CORS_ORIGIN=http://localhost:3000

# ---- PostgreSQL ----
# host = the service name from docker-compose (postgres), not localhost
DATABASE_URL=postgres://devvault:devvault@postgres:5432/devvault

# ---- Redis ----
REDIS_URL=redis://redis:6379
терминал · dev-vault-server/bash
# создать рабочий .env из шаблона
cp .env.example .env
про том node_modules

Строка /app/node_modules без хостового пути — анонимный том. Он «прикрывает» node_modules из образа, чтобы твоя папка с хоста их не затёрла. Без этого контейнер часто падает с «module not found».

Клиент — dev-vault-client frontend

Репа dev-vault-client — Vite + React (npm create vite). Файлы проще серверных: базы и кэша фронту не нужно. Но есть nginx.conf — для раздачи статики в проде.

# dev-vault-client/ ├── src/ # уже есть (Vite + React) ├── package.json # уже есть ├── Dockerfile # ← добавляем ├── .dockerignore # ← добавляем ├── docker-compose.yml # ← добавляем ├── nginx.conf # ← добавляем (SPA-раздача в проде) ├── .env.example # ← шаблон, коммитится └── .env # ← cp .env.example .env (в .gitignore)

Dockerfile

Каркас тот же, но финал другой: бэк в проде крутит Node-процесс, а фронт компилируется в статику и раздаётся через Nginx (Node в проде не нужен).

dev-vault-client/Dockerfiledockerfile
# syntax=docker/dockerfile:1

# ---- base: shared by all stages ----
FROM node:24-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- deps: install dependencies (cacheable layer) ----
FROM base AS deps
RUN npm ci

# ---- dev: Vite dev server with hot-reload ----
FROM deps AS dev
COPY . .
EXPOSE 3000
# --host — reachable from outside the container; --port 3000 — front on 3000 (back on 3030)
CMD ["npm", "run", "dev", "--", "--host", "--port", "3000"]

# ---- build: compile static assets into dist/ ----
FROM deps AS build
COPY . .
RUN npm run build

# ---- prod: serve static files via Nginx, no Node ----
FROM nginx:alpine AS prod
COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

docker-compose.yml

dev-vault-client/docker-compose.ymlyaml
services:
  web:
    build:
      context: .
      target: dev            # use the dev stage from the Dockerfile
    command: npm run dev -- --host --port 3000
    ports:
      - "3000:3000"
    env_file: .env
    volumes:
      - .:/app               # mount host code into the container for hot-reload
      - /app/node_modules    # but keep node_modules from the image

nginx.conf

Нужен для прод-стадии: раздаёт собранную статику и отдаёт index.html на любой путь — иначе клиентский роутинг ломается при обновлении страницы.

dev-vault-client/nginx.confnginx
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # SPA: serve index.html for any path — otherwise client-side routing breaks on F5
    location / {
        try_files $uri $uri/ /index.html;
    }
}

ключевой момент: React крутится в браузере, а не в контейнере

Контейнер web только отдаёт собранный JS. Сам код React исполняется в твоём браузере — и запросы к API уходят из браузера на localhost:3030. Поэтому фронт и знать не знает про внутреннюю сеть Docker: он бьёт по опубликованному порту.

Корневой compose — весь стек одной командой оркестратор

Теперь самое приятное. В рабочей папке dev-vault/ — крошечный compose, который ничего не дублирует, а через include подключает compose-файлы обеих реп. Одна команда отсюда поднимает всё: сервер, клиент, базу и кэш.

dev-vault/docker-compose.ymlyaml
# Root compose: brings up the whole dev-vault stack with one command.
# Reuses each project's own compose file — no duplicated config here.
#   docker compose up        (from this directory)
include:
  - dev-vault-server/docker-compose.yml
  - dev-vault-client/docker-compose.yml
что даёт include

Compose берёт указанные файлы и сливает их в один проект. Все четыре сервиса оказываются в общей сети, поднимаются и гасятся вместе. При этом каждая репа остаётся самодостаточной — можно зайти в неё и запустить только её docker compose up.

Запуск и проверка поехали

Из корневой папки dev-vault/ — одна команда собирает образы и поднимает весь стек:

терминал · dev-vault/bash
# собрать образы и поднять весь стек (логи в этом же окне)
docker compose up --build
первый запуск — дольше, это норма

В первый раз Docker скачивает образы (Node, Postgres, Redis, Nginx) и собирает твои — пара минут. Дальше секунды: образы кэшированы.

Когда логи успокоятся — открывай обе точки:

браузерurl
http://localhost:3030    # сервер (Nest) → Hello World!
http://localhost:3000    # клиент (React) → страница Vite

Теперь проверь hot-reload на любой из сторон — правка в коде подхватывается без перезапуска:

  1. Сервер: поменяй строку в dev-vault-server/src/app.service.ts (метод getHello()).
  2. Клиент: поменяй текст в dev-vault-client/src/App.tsx.
  3. Сохрани — контейнер сам пересоберётся, обнови вкладку, текст изменился.
на macOS hot-reload не сработал?

На Mac события файловой системы не всегда доходят в контейнер через bind-mount. Для сервера лечится поллингом — в dev-vault-server/tsconfig.json добавь на верхний уровень:

dev-vault-server/tsconfig.jsonjson
"watchOptions": {
  "watchFile": "priorityPollingInterval"
}
проверь себя

Четыре контейнера в docker compose ps со статусом up, localhost:3030 отдаёт Hello World, localhost:3000 — страницу React, правки кода подхватываются. Если так — этап 01 выполнен.

CORS — почему сервер знает про localhost:3000

Фронт (localhost:3000) и бэк (localhost:3030) — разные origin (отличается порт). Браузер по умолчанию блокирует запросы между разными origin — это защита CORS. Чтобы фронту разрешили звать API, сервер должен явно сказать «этому адресу можно». Поэтому в серверном .env и появился CORS_ORIGIN=http://localhost:3000 — сервер прочитает его и разрешит. Саму настройку CORS в коде Nest включим на этапе 06; сейчас достаточно понимать, зачем эта переменная.

Частые вопросы

Что делает include в корневом compose?

Берёт указанные compose-файлы и сливает их сервисы в один проект — как будто ты написал их в одном файле. Все контейнеры попадают в общую сеть и управляются вместе.

Плюс: ноль дублирования. Каждая репа хранит свой compose, а корневой лишь ссылается на них. Правишь сервер — правишь только его файл.

Почему две репы, а не один общий проект?

Сервер и клиент независимы: свои коммиты, свой деплой, разные жизненные циклы, могут вести разные люди. Их не связывают импортом — только по HTTP через API.

Корневой compose — это удобство локальной разработки, чтобы поднять всё одной командой. На сервере (этап 02) каждый едет сам.

Зачем серверу переменная CORS_ORIGIN?

Фронт на :3000 и бэк на :3030 — разные origin. Браузер блокирует запросы между разными origin, пока сервер явно не разрешит. CORS_ORIGIN и есть этот «разрешённый адрес фронта».

Саму логику CORS в Nest включим на этапе 06 — сейчас переменная просто лежит наготове.

Контейнер — это виртуальная машина?

Нет. ВМ тащит целую гостевую ОС — гигабайты и старт в минуты. Контейнер делит ядро твоей ОС и изолирует только процесс. Отсюда мегабайты и старт за секунды.

Поэтому нормально держать четыре контейнера там, где четыре виртуалки были бы расточительством.

Что с данными при docker compose down?

Контейнеры удаляются — они одноразовые. Но тома (pgdata, redisdata) остаются, поэтому после down и up база с теми же данными.

Сотрёт только docker compose down -v (флаг -v = удалить тома) — это «полный сброс базы».

Зачем отдельная стадия deps в Dockerfile?

Она ставит зависимости (npm ci) один раз, а стадии dev и build её переиспользуют. Правишь код (но не package.json) — Docker берёт зависимости из кэша и не переустанавливает. Сборка быстрее.

Нужно пересобирать образ после каждой правки кода?

В режиме разработки — нет. Код примонтирован томом, а dev-сервер (start:dev у бэка, Vite у фронта) следит за изменениями и пересобирает на лету.

Пересборка (up --build) нужна, только когда меняешь зависимости или сам Dockerfile.

Почему у сервера порт 3030, а не 3000?

Порт 3000 занят фронтом (типичный порт React-стека). Чтобы фронт и бэк не конфликтовали на одной машине, бэк уводим на 3030. Оба порта проброшены на хост и не мешают друг другу.

что у тебя теперь есть

Воспроизводимое окружение: сервер + клиент + Postgres + Redis поднимаются одной командой из корня, код перезагружается на лету, данные живут в томах. Ровно эти же compose-файлы станут основой боевого стека на сервере (этап 02).

Перенос на сервер — превью

Сила Docker в том, что те же compose-файлы поедут на сервер почти без изменений. На твоём Hetzner в общих чертах: SSH → установить Docker → забрать репы → прод-.env → docker compose up -d.

но «боевой» сервер — это больше

По-настоящему рабочий сервер — это ещё вход по ключу, firewall, Nginx, HTTPS, автозапуск и — важно — закрытые наружу порты базы. Всё по шагам под твой Hetzner — в Уроке 02 «Боевой сервер» →.
