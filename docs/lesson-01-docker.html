Локальная среда в Docker

dev-vault — не монолит, а четыре независимых репы: API, клиент, админка и инфраструктура. Каждое приложение разрабатывается и деплоится само по себе, а фронты ходят к API по HTTP — это разные origin, поэтому нужен CORS. В этом уроке — локальная разработка: Docker-файлы каждого приложения и два способа поднять стек.

~60 мин Docker · Compose этап 01 плана нужен Docker Desktop (или только Node)

Проект: четыре репы архитектура

Разделение по репозиториям — не прихоть: у каждого свой цикл, свои коммиты, свой деплой. Связывает их только HTTP-контракт (API), а не общий код.

РепозиторийРольЛокально
dev-vault-serverBackend API (NestJS)localhost:3030
dev-vault-clientКлиентский SPA (React + Vite)localhost:3000
dev-vault-adminАдминка SPA (React + Vite)localhost:3001
dev-vault-infraОркестрация (Compose + Caddy)—
# репы лежат рядом друг с другом dev-vault/ ├── dev-vault-server/ # API (Nest) + Postgres + Redis ├── dev-vault-client/ # публичный фронт ├── dev-vault-admin/ # админка └── dev-vault-infra/ # compose.dev.yml · compose.prod.yml · Caddyfile

Что построим локально результат этапа

Пять контейнеров. Фронты стучатся к API по его адресу с хоста — localhost:3000 и localhost:3030 это разные origin, поэтому API должен явно разрешить их через CORS_ORIGIN.

браузер │ localhost:3000 клиент ──запрос к API──┐ │ localhost:3001 админка ──запрос к API──┤ cross-origin → CORS ▼ ▼ (статика/dev-сервер Vite) localhost:3030 API (Nest) │ ┌─────────┴─────────┐ postgres :5432 redis :6379
важно про этот урок

Код не подключаем к базе (это этап 05, Prisma) и не пишем логику. Цель — рабочее локальное окружение и понимание, как устроены Docker-файлы каждого приложения.

Мини-теория: 4 слова Docker

📦

образ (image)

Замороженный слепок: код + среда + зависимости. Как установочный диск — сам не работает, но с него запускают.

▶️

контейнер (container)

Запущенный образ — живой процесс. Из одного образа можно поднять хоть десять контейнеров.

💾

том (volume)

Место, где данные переживают перезапуск контейнера. Без него база потеряет всё при down.

🕸️

сеть (network)

Compose соединяет контейнеры в сеть, где они находят друг друга по имени сервиса — так API видит базу как postgres.

🔗 узнать больше

Глубже про эти четыре слова — Docker: ментальная модель (образ vs контейнер, слои и кэш с интерактивом). Каждую строчку Dockerfile и compose разбирает справочник. А про базу и кэш из этого урока — PostgreSQL и Redis.

Сервер — dev-vault-server API + база + кэш

Dockerfile

Multi-stage: база → deps (зависимости одним слоем) → dev и build её переиспользуют → prod — минимальный образ под non-root (USER node). Один файл обслуживает и локалку, и прод.

dev-vault-server/Dockerfiledockerfile
# syntax=docker/dockerfile:1

# ---- base: shared by all stages ----
FROM node:24-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- deps: install dependencies (cacheable layer) ----
FROM base AS deps
RUN npm ci

# ---- dev: dev dependencies + hot-reload ----
FROM deps AS dev
COPY . .
CMD ["npm", "run", "start:dev"]

# ---- build: production build ----
FROM deps AS build
COPY . .
RUN npm run build
RUN npm prune --omit=dev

# ---- prod: minimal image for the server ----
FROM node:24-alpine AS prod
WORKDIR /app
ENV NODE_ENV=production

COPY --from=build --chown=node:node /app/node_modules ./node_modules
COPY --from=build --chown=node:node /app/dist ./dist
COPY --from=build --chown=node:node /app/package*.json ./

USER node
EXPOSE 3030
CMD ["node", "dist/main.js"]

docker-compose.yml — локальная разработка

Поднимает API + Postgres + Redis. Порты БД открыты наружу — это удобно локально (psql, GUI). В проде так делать нельзя, там свой файл (урок 02).

dev-vault-server/docker-compose.ymlyaml · dev
services:
  app:
    build:
      context: .
      target: dev            # use the dev stage from the Dockerfile
    command: npm run start:dev
    ports:
      - "3030:3030"
    env_file: .env
    volumes:
      - .:/app               # mount host code into the container for hot-reload
      - /app/node_modules    # but keep node_modules from the image
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_started

  postgres:
    image: postgres:18-alpine
    environment:
      POSTGRES_USER: devvault
      POSTGRES_PASSWORD: devvault
      POSTGRES_DB: devvault
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devvault"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:8-alpine
    ports:
      - "6379:6379"
    volumes:
      - redisdata:/data

volumes:
  pgdata:
  redisdata:
две ловушки

1. В postgres:18 путь данных — /var/lib/postgresql (без /data, как было в 17). 2. Из контейнера app база доступна как postgres:5432, а не localhost: внутри контейнера localhost — это он сам, а имя сервиса из compose = адрес соседа.

.env.example

Шаблон коммитим, рабочий .env — нет. CORS_ORIGIN — белый список origin'ов фронтов, которым разрешено звать API.

dev-vault-server/.env.exampleenv
# ---- Application ----
# Port must match the port in docker-compose.yml and EXPOSE in the Dockerfile
PORT=3030
NODE_ENV=development

# Frontend origins allowed to call the API (CORS) — comma-separated allowlist.
# Dev: client (3000) + admin (3001). Prod: https://app.YOUR_DOMAIN,https://admin.YOUR_DOMAIN
CORS_ORIGIN=http://localhost:3000,http://localhost:3001

# ---- PostgreSQL ----
# host = the service name from docker-compose (postgres), not localhost
DATABASE_URL=postgres://devvault:devvault@postgres:5432/devvault

# ---- Redis ----
REDIS_URL=redis://redis:6379

Клиент — dev-vault-client React SPA

Vite + React. Dockerfile тот же по форме, но финал другой: в проде фронт — это статика, которую раздаёт Nginx, а не Node. Ключевая деталь — ARG VITE_API_URL на стадии сборки.

Dockerfile

dev-vault-client/Dockerfiledockerfile
# syntax=docker/dockerfile:1

# ---- base: shared by all stages ----
FROM node:24-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- deps: install dependencies (cacheable layer) ----
FROM base AS deps
RUN npm ci

# ---- dev: Vite dev server with hot-reload ----
FROM deps AS dev
COPY . .
EXPOSE 3000
# --host — reachable from outside the container; --port 3000 — front on 3000 (back on 3030)
CMD ["npm", "run", "dev", "--", "--host", "--port", "3000"]

# ---- build: compile static assets into dist/ ----
# VITE_API_URL is baked into the bundle at build time, so it must be passed
# per environment via --build-arg (dev: localhost, prod: https://api.YOUR_DOMAIN).
FROM deps AS build
ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL
COPY . .
RUN npm run build

# ---- prod: serve static files via Nginx, no Node ----
FROM nginx:alpine AS prod
COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
почему VITE_API_URL — это build-arg, а не runtime-переменная

Vite запекает VITE_-переменные в бандл на этапе сборки. После сборки их уже не поменять — в браузер уезжает готовый JS. Поэтому локально значение берётся из .env, а для прода передаётся при сборке образа: --build-arg VITE_API_URL=https://api.твой-домен (это делает compose.prod.yml, урок 02).

docker-compose.yml, vite.config.ts, .env.example

dev-vault-client/docker-compose.ymlyaml · dev
services:
  web:
    build:
      context: .
      target: dev            # use the dev stage from the Dockerfile
    command: npm run dev -- --host --port 3000
    ports:

import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

// https://vite.dev/config/
export default defineConfig({
  plugins: [react()],
  // Client runs on 3000 locally (admin uses 3001, backend 3030).
  server: {
    port: 3000,
    strictPort: true,
  },

    port: 3000,
  },

dev-vault-client/.env.exampleenv
# ---- Client (Vite) ----
# Vite variables MUST be prefixed with VITE_ — otherwise Vite won't expose them to the code.
# Note: VITE_ variables are embedded in the built bundle and visible in the browser — no secrets here.

# Backend API address (dev-vault-server listens on 3030).
# Dev only — used by `npm run dev`/`npm run build` locally.
# In prod the value is injected at image build via --build-arg VITE_API_URL=https://api.YOUR_DOMAIN.
VITE_API_URL=http://localhost:3030

Админка — dev-vault-admin React SPA

Второй фронт к тому же API. Dockerfile, nginx.conf и .env.example — идентичны клиенту. Отличается только compose: имя сервиса admin и проброс порта 3001:3000 — внутри контейнера Vite слушает свой обычный 3000, а на хосте занимает 3001, чтобы не конфликтовать с клиентом.

dev-vault-admin/docker-compose.ymlyaml · dev
services:
  admin:
    build:
      context: .
      target: dev            # use the dev stage from the Dockerfile
    command: npm run dev -- --host --port 3000
    ports:
      - "3001:3000"          # host 3001 -> container 3000 (client uses 3000)
    env_file: .env
    volumes:
      - .:/app               # mount host code into the container for hot-reload
      - /app/node_modules    # but keep node_modules from the image

Запуск: два способа поехали

Сервисы независимы, поэтому и запускать их можно по-разному. Выбирай под задачу.

A. Без Docker — когда работаешь над одним приложением

Самый частый режим: правишь один фронт, а API уже поднят. Каждое приложение стартует само:

терминалbash
# terminal 1
cd dev-vault-server && npm run start:dev    # http://localhost:3030
# terminal 2
cd dev-vault-client && npm run dev          # http://localhost:3000
# terminal 3
cd dev-vault-admin  && npm run dev          # http://localhost:3001

B. Одной командой — весь стек в Docker

В dev-vault-infra лежит compose.dev.yml, который ничего не дублирует — он просто include-ит compose-файлы трёх приложений:

dev-vault-infra/compose.dev.ymlyaml · dev
# Local development stack: brings up all three apps + Postgres + Redis with one
# command. Reuses each app's own docker-compose.yml — no duplicated config here.
# Expects the app repos cloned side by side next to this one. Run from here:
#   docker compose -f compose.dev.yml up
include:
  - ../dev-vault-server/docker-compose.yml
  - ../dev-vault-client/docker-compose.yml
  - ../dev-vault-admin/docker-compose.yml
терминал · dev-vault-infra/bash
# перед первым запуском в каждой репе: cp .env.example .env
docker compose -f compose.dev.yml up
dev и prod — намеренно разные файлы

compose.dev.yml — для локальной работы, compose.prod.yml — для сервера. Вместе их не запускают. Общее у них — Dockerfile каждого приложения: одна и та же multi-stage сборка, просто разные стадии (dev / prod).

Открывай:

браузерurl
http://localhost:3030    # API (Nest)
http://localhost:3000    # клиент (React)
http://localhost:3001    # админка (React)
проверь себя

API отвечает на localhost:3030, клиент и админка открываются на 3000/3001, их запросы к API проходят (в консоли браузера нет ошибок CORS), правки кода подхватываются без ручного перезапуска.

на macOS hot-reload не сработал?

На Mac события файловой системы не всегда доходят в контейнер через bind-mount. Для сервера лечится поллингом — в dev-vault-server/tsconfig.json добавь на верхний уровень "watchOptions": { "watchFile": "priorityPollingInterval" } и перезапусти.

CORS — почему он нужен и никуда не денется

Клиент (:3000), админка (:3001) и API (:3030) — разные origin. Браузер по умолчанию блокирует запросы между разными origin. Чтобы фронтам разрешили звать API, сервер обязан явно перечислить их адреса — это и есть CORS_ORIGIN.

Так и в проде: приложения деплоятся отдельно, на разных поддоменах (api., app., admin.) — снова разные origin, снова CORS (только список уже боевой). Поэтому CORS — сквозной механизм dev → prod, а не локальный костыль. Саму настройку в коде Nest включим на этапе 06.

🔗 узнать больше

Как эти разные origin и поддомены выглядят в общей картине — на карте системы (переключи dev/prod). А кто в проде раздаёт запросы по поддоменам — Nginx: reverse-proxy.

Частые вопросы

Почему четыре репы, а не один проект?

У каждого приложения свой жизненный цикл: свои коммиты, свои зависимости, свой деплой, могут вести разные люди. Они не связаны общим кодом — только HTTP-контрактом (API).

Четвёртая репа, dev-vault-infra, не содержит приложения — только оркестрацию: как всё поднять локально и как развернуть в проде.

Зачем CORS, если можно было бы проксировать через один origin?

Потому что приложения независимы и работают отдельно: обычно ты правишь один фронт, а API уже развёрнут (у тебя на хосте или на деве). Фронт зовёт его напрямую — это другой origin.

Прокси «в один origin» работал бы только когда всё крутится вместе — а это не наш режим. Cross-origin + CORS одинаково работает и локально, и в проде.

Почему VITE_API_URL нельзя задать при запуске контейнера?

Vite запекает VITE_-переменные в бандл при сборке. В браузер уезжает готовый JS — менять переменную в рантайме уже нечему.

Поэтому для прода адрес API передают в сборку образа: --build-arg VITE_API_URL=.... Меняешь домен — пересобираешь образ.

Почему у админки порт 3001:3000, а не 3001:3001?

Внутри контейнера Vite слушает свой обычный 3000 — контейнеры изолированы, конфликта там нет. А на хосте порт 3000 уже занят клиентом, поэтому админку выставляем на 3001.

Слева — порт хоста, справа — порт внутри контейнера. Классический приём.

Что с данными при docker compose down?

Контейнеры удаляются, но тома (pgdata, redisdata) остаются — база после up будет с теми же данными. Сотрёт только down -v.

Зачем стадия deps в Dockerfile?

Она ставит зависимости один раз, а dev и build её переиспользуют. Правишь код (но не package.json) — Docker берёт слой из кэша, сборка быстрее.

что у тебя теперь есть

Локальное окружение четырёх реп: API + база + кэш и два фронта, hot-reload, cross-origin + CORS. Те же Dockerfile (стадия prod) поедут на сервер — через compose.prod.yml и Caddy. Это Урок 02.
