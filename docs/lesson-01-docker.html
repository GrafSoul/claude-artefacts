Локальная среда в Docker

Заворачиваем пустой Nest-проект dev-vault в контейнер и поднимаем рядом PostgreSQL и Redis — всё одной командой, с автоперезагрузкой при правке кода. К концу урока у тебя воспроизводимое окружение, одинаковое у тебя на машине и потом на сервере.

~50 мин Docker · Compose шаги плана 1.1–1.3 нужен установленный Docker Desktop

Что построим результат этапа

Полная картина: четыре контейнера в одной сети Docker — web (React), app (Nest) и рядом база с кэшем. В этом уроке поднимаем бэкенд-часть (app + postgres + redis); web живёт в своей репе и встаёт в ту же сеть тем же способом — показан, чтобы ты видел систему целиком.

Бэкенд-стек: три контейнера, но одно приложение

Частый ступор: «почему три контейнера, это же одно приложение?». Речь про бэкенд-стек (app + база + кэш) — разберём, это основа, на которой держится всё дальше.

Три контейнера — это не три копии твоего кода. Твоё приложение среди них одно — app. Два других — готовые чужие программы, которые ты не пишешь, а просто используешь:

appтвой Nest — dev-vaultтвой код
postgresсервер базы данныхчужой образ
redisсервер кэшачужой образ

Аналогия: розетка-удлинитель. Приложение + база + кэш — три разных прибора, воткнутых в одну сеть. Не три одинаковых, а три разных, работающих вместе.

«изолированы» ≠ «разъединены»

Контейнеры изолированы как процессы (свой Node, свои библиотеки, свой рестарт), но соединены сетью Docker и свободно общаются. Изоляция здесь — про независимость, а не про стену.

Compose оборачивает всё в одну сущность

Отдельные контейнеры неудобно запускать по одному. docker-compose.yml — это как package.json, только для инфраструктуры: один манифест всех частей и того, как они связаны. Compose группирует их в проект (по имени папки — dev-vault) с общей сетью и общим жизненным циклом.

docker-compose.yml # одно описание │ docker compose up ─────────────┐ │ │ ┌────────────┼─────────────────┼──────────┐ проект «dev-vault» │ app postgres redis │ общая сеть, общий up/down └──────────────────────────────────────────┘

Снаружи — одна сущность (описал одним файлом, поднял одной командой), внутри — независимые части. Поэтому позже одной строкой можно размножить только приложение: docker compose up --scale app=3 — три копии app, но по-прежнему одна база и один Redis. Будь это монолитный контейнер — так бы не вышло.

React тоже в Docker — из своей репы

Код фронта — отдельная репа (React ты знаешь, учить не будем). Но в Docker-картине web — это ещё один контейнер рядом с бэком, как на схеме «Что построим» выше. Значит система целиком живёт в Docker: web + app + база + кэш. Фронт контейнеризуется тем же способом — свой Dockerfile, те же команды.

одна сеть Docker ┌──────────────────────────────────────────────┐ │ web ──HTTP──► app ──► postgres / redis │ │ React Nest база / кэш │ └──────────────────────────────────────────────┘ репа frontend └──── репа dev-vault ────┘

Локально: контейнеры в одной Docker-сети; фронт зовёт бэк по localhost:3000 — как ты сейчас дёргаешь любой API. В прод (этап 02) Nginx ставит оба под один домен: app.домен → фронт, /api → бэк.

две репы, но одна Docker-картина

Репы разные — разные деплои и жизненные циклы. А Docker и сеть общие: контейнеры видят друг друга. Единственная связь фронт↔бэк — API-контракт (этап 06); CORS и токены тоже там.

Мини-теория: 4 слова Docker

Только то, без чего дальше будет непонятно. Аналогии намеренно бытовые.

📦

образ (image)

Замороженный слепок: код + среда + зависимости. Как установочный диск — сам не работает, но с него запускают.

▶️

контейнер (container)

Запущенный образ — живой процесс. Из одного образа можно поднять хоть десять контейнеров.

💾

том (volume)

Место, где данные переживают перезапуск контейнера. Без него база потеряет всё при down.

🕸️

сеть (network)

Compose соединяет контейнеры в одну сеть, где они находят друг друга по имени сервиса, а не по IP.

Шаг 1 — Dockerfile план 1.1

Multi-stage: один файл описывает и лёгкий боевой образ, и стадию для разработки. Сейчас нам нужна стадия dev, но prod пусть будет сразу — пригодится на этапе 03.

Dockerfiledockerfile
# ---- база: общая для всех стадий ----
FROM node:22-alpine AS base
WORKDIR /app
COPY package*.json ./

# ---- dev: с dev-зависимостями и hot-reload ----
FROM base AS dev
RUN npm ci
COPY . .
CMD ["npm", "run", "start:dev"]

# ---- build: собираем прод-сборку ----
FROM base AS build
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --omit=dev

# ---- prod: минимальный образ для сервера ----
FROM node:22-alpine AS prod
WORKDIR /app
ENV NODE_ENV=production
COPY --from=build /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
COPY --from=build /app/package*.json ./
EXPOSE 3000
CMD ["node", "dist/main.js"]

почему multi-stage

Стадия dev монтирует твой код и следит за изменениями. Стадия prod не содержит ни dev-зависимостей, ни исходников — только собранный dist. Финальный образ выходит в разы меньше и безопаснее.

Dockerfile фронта — для сравнения (другая репа)

У React-морды свой Dockerfile, и он другой — потому что задача другая. Бэк в проде крутит живой Node-процесс; фронт компилируется в статику, которую раздаёт Nginx (Node в проде уже не нужен). Каркас похож, финал — принципиально разный:

Dockerfile · репа frontend (React)dockerfile
# ---- dev: Vite dev-сервер с hot-reload ----
FROM node:22-alpine AS dev
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
EXPOSE 5173
# --host — чтобы Vite был виден снаружи контейнера
CMD ["npm", "run", "dev", "--", "--host"]

# ---- build: собираем статику в папку dist/ ----
FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# ---- prod: отдаём статику через Nginx, без Node ----
FROM nginx:alpine AS prod
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
«совместимость» тут не про Dockerfile

Файлы фронта и бэка не обязаны совпадать — даже версия Node может отличаться. Они общаются по HTTP, а не разделяют код. Связывают их три вещи: адрес API (env-переменная VITE_API_URL), сам API-контракт и CORS на бэке. Подробно — этап 06.

Шаг 2 — docker-compose.yml план 1.2

Описываем весь стек в одном файле: приложение + база + кэш, их тома, порты и связи. depends_on с проверкой здоровья гарантирует, что app стартует, только когда Postgres реально готов принимать соединения.

откуда берутся Postgres и Redis — их ставить НЕ надо

Главный момент: базу и кэш не нужно устанавливать вручную. Строки image: postgres:17-alpine и image: redis:7-alpine говорят Docker: «скачай готовый образ из реестра Docker Hub». При первом docker compose up Docker сам их подтянет и запустит. На машине тебе нужен только Docker — ни Postgres, ни Redis отдельно ставить не требуется.

docker-compose.ymlyaml

частая ловушка

Из контейнера app база доступна как postgres:5432, а не localhost:5432. localhost внутри контейнера — это сам контейнер. Имя сервиса из compose = сетевой адрес соседа.

про том node_modules

Строка /app/node_modules без хостового пути — это анонимный том. Он «прикрывает» node_modules из образа, чтобы твоя папка с хоста их не затёрла. Без этого приёма контейнер часто падает с «module not found».

Шаг 3 — запуск и hot-reload план 1.3

Из папки dev-vault — одна команда, которая соберёт образ и поднимет все три контейнера:

терминал · dev-vault/bash
# собрать образы и поднять весь стек (логи в этом же окне)
docker compose up --build
первый запуск — дольше, это норма

В первый раз Docker скачивает образы (Node, Postgres, Redis) и собирает твой — пара минут. Дальше секунды: образы уже кэшированы на диске.

Когда в логах появится строка Nest «Nest application successfully started» — открывай:

браузерurl
http://localhost:3000

Увидишь Hello World! — приложение из контейнера отвечает наружу. Теперь проверим hot-reload:

  1. Открой src/app.service.ts и поменяй строку в методе getHello() (там лежит 'Hello World!').
  2. Сохрани файл — в логах контейнера Nest сам пересоберётся за секунду.
  3. Обнови вкладку — текст изменился, хотя ты ничего не перезапускал руками.
проверь себя

Три контейнера в docker compose ps со статусом up, приложение отвечает на localhost:3000, правка кода подхватывается без ручного перезапуска. Если так — этап 01 выполнен.

Частые вопросы новичков

Вопросы, на которых спотыкаются почти все в начале. Разверни нужный.

Контейнер — это виртуальная машина?

Нет. Виртуальная машина тащит целую гостевую операционную систему — это гигабайты и старт в минуты. Контейнер делит ядро твоей ОС и изолирует только процесс с его файлами и зависимостями.

Итог: контейнер весит мегабайты и стартует за секунды. Поэтому нормально держать три контейнера там, где три виртуалки были бы расточительством.

Что с данными при docker compose down?

Контейнеры удаляются — они одноразовые. Но тома (pgdata, redisdata) остаются, поэтому база после down и повторного up будет с теми же данными.

Данные сотрёт только явный docker compose down -v (флаг -v = удалить тома). Это твой «полный сброс базы».

build, up, run — в чём разница?

build — собрать образ из Dockerfile. run — запустить один контейнер из образа. up (это Compose) — собрать, если нужно, и поднять весь стек сразу.

В повседневной работе ты почти всегда используешь docker compose up, а «голые» build/run — редко и точечно.

Почему образы с приставкой -alpine?

Alpine — минимальный дистрибутив Linux (~5 МБ). Образ на нём меньше: быстрее качается, быстрее деплоится, меньше поверхность для уязвимостей.

Изредка какой-то нативной библиотеки в alpine не хватает — тогда берут обычный node:22. Для старта alpine — разумный дефолт.

Зачем COPY package*.json идёт до COPY . .?

Каждая инструкция Dockerfile — это слой, и Docker их кэширует. Если сначала скопировать только package.json и поставить зависимости, а код копировать после — то при правке кода (но не зависимостей) Docker переиспользует кэш и не переустанавливает npm заново.

Результат: пересборка образа ускоряется в разы. Это не случайный порядок, а приём оптимизации.

Зачем depends_on вместе с healthcheck?

depends_on задаёт порядок старта, а condition: service_healthy заставляет app ждать, пока Postgres реально готов принимать соединения — а не просто «процесс запустился».

Без этого приложение может стартовать раньше базы, не достучаться до неё и упасть на старте.

Почему localhost внутри контейнера не находит базу?

Внутри контейнера localhost — это сам контейнер, а не твоя машина и не соседний контейнер. База живёт в другом контейнере.

Её адрес — имя сервиса из compose: postgres:5432. Compose сам поднимает внутренний DNS, где имена сервисов работают как адреса.

Нужно пересобирать образ после каждой правки кода?

В режиме разработки — нет. Код примонтирован томом с хоста, а start:dev следит за изменениями и пересобирает Nest на лету внутри контейнера.

Пересборка (docker compose up --build) нужна, только когда меняешь зависимости (package.json) или сам Dockerfile.

Где физически лежат данные базы?

В томе Docker (pgdata), которым управляет сам Docker — не в папке твоего проекта. Список томов: docker volume ls.

Именно поэтому данные переживают удаление и пересоздание контейнера: контейнер одноразовый, том — нет.

что у тебя теперь есть

Воспроизводимое окружение: dev-vault + Postgres + Redis поднимаются одной командой, код перезагружается на лету, данные базы живут в томе. Ровно этот же docker-compose.yml станет основой боевого стека на сервере (этап 02).

Перенос на сервер — превью

Сила Docker в том, что тот же docker-compose.yml, что работает у тебя локально, поедет на сервер почти без изменений. На твоём Hetzner в общих чертах так:

  1. Подключиться к серверу по SSH.
  2. Установить Docker — так же, как локально.
  3. Забрать проект на сервер (git clone / git pull).
  4. Положить прод-.env — другие пароли, реальный домен.
  5. docker compose up -d — и стек уже крутится на сервере.
но «боевой» сервер — это больше

По-настоящему рабочий сервер — это ещё вход по ключу, firewall, Nginx перед приложением, HTTPS, автозапуск после перезагрузки. Это отдельная большая тема — урок 02 «Боевой сервер (VPS)», как раз под твой Hetzner: там разберёмся с железом по шагам. Здесь главное усвоить: локальный и серверный стек — один и тот же compose, меняется в основном конфиг.
