Docker & Compose построчно

Каждая настройка из твоих файлов netroom — с человеческим объяснением: что это, зачем и где подвох. Слева — строка, справа — расшифровка. Это спутник к Уроку 01: там ты копируешь файлы, здесь понимаешь каждую строку.

Как читать Dockerfile

Dockerfile — это рецепт образа, читается сверху вниз. Каждая инструкция создаёт слой, который Docker кэширует: если слой и всё до него не изменились — он берётся из кэша, а не пересобирается. Поэтому порядок инструкций влияет на скорость сборки. Ещё один приём — multi-stage: в одном файле несколько стадий (FROM ... AS имя), и в финальный образ попадает только то, что реально нужно.

Dockerfile сервера

Backend (netroom-server/Dockerfile). Четыре стадии: общая база, установка зависимостей, разработка/сборка, лёгкий прод.

netroom-server/Dockerfile
# syntax=docker/dockerfile:1
Директива парсера: использовать актуальный синтаксис Dockerfile (версия 1). Ставится самой первой строкой, даёт доступ к свежим возможностям сборки.
FROM node:24-alpine AS base
Стадия 1 — база. FROM — с какого образа стартуем. node:24 — официальный образ Node.js 24. alpine — минимальный Linux (~5 МБ) внутри. AS base — имя стадии, чтобы дальше на неё ссылаться.
WORKDIR /app
Рабочая папка внутри контейнера. Все следующие команды идут относительно /app. Если папки нет — создастся.
COPY package*.json ./
COPY копирует файлы с твоей машины в образ. package*.json — маска: package.json + package-lock.json. ./ — в рабочую папку. Копируем только манифесты (не весь код), чтобы слой с зависимостями кэшировался отдельно.
FROM base AS deps RUN npm ci
Стадия 2 — зависимости. Наследует base. RUN выполняет команду при сборке образа. npm ci — чистая установка строго по lock-файлу (быстрее и воспроизводимее npm install). Слой переиспользуется, пока не менялся package.json.
FROM deps AS dev COPY . . CMD ["npm", "run", "start:dev"]
Стадия 3 — разработка. Поверх deps (зависимости уже стоят). COPY . . — весь код в образ (после установки зависимостей, чтобы правки кода не сбрасывали их кэш). CMD — команда при запуске контейнера (не при сборке); форма-массив — без обёртки shell.
FROM deps AS build COPY . . RUN npm run build RUN npm prune --omit=dev
Стадия 4 — сборка прода. npm run build компилирует TypeScript в JS (папка dist/). npm prune --omit=dev выкидывает dev-зависимости — в проде они не нужны, образ меньше.
FROM node:24-alpine AS prod WORKDIR /app ENV NODE_ENV=production
Стадия 5 — финальный прод-образ. Начинается с чистого Node — без исходников и dev-инструментов предыдущих стадий. ENV — переменная окружения, зашитая в образ; NODE_ENV=production включает прод-режим у библиотек.
COPY --from=build --chown=node:node \ /app/node_modules ./node_modules COPY --from=build --chown=node:node \ /app/dist ./dist COPY --from=build --chown=node:node \ /app/package*.json ./
--from=build — копируем не с хоста, а из стадии build. Берём только готовое: node_modules (уже без dev), dist, манифесты. --chown=node:node — сразу назначаем владельцем файлов пользователя node, а не root.
USER node
Дальше контейнер работает под непривилегированным пользователем node, а не root. Если приложение взломают — прав у процесса меньше. Базовая безопасность.
EXPOSE 3030
Документирует, что контейнер слушает порт 3030. Сам порт наружу НЕ открывает — это делает ports в compose. Это подсказка для читателя и инструментов.
CMD ["node", "dist/main.js"]
Команда запуска прод-контейнера: просто node на собранном файле — без npm и nest-cli, которых в проде уже нет.

Dockerfile клиента — что отличается

Frontend (netroom-client/Dockerfile) устроен так же (база → deps → dev → build), но финал другой: React собирается в статику, которую в проде раздаёт Nginx, а не Node.

netroom-client/Dockerfile · только отличия
EXPOSE 3000 CMD ["npm", "run", "dev", "--", "--host", "--port", "3000"]
dev-стадия. --host — Vite слушает 0.0.0.0, иначе снаружи контейнера к нему не достучаться. --port 3000 — фиксируем порт фронта (бэк на 3030). -- отделяет аргументы npm от аргументов Vite.
FROM deps AS build ARG VITE_API_URL ENV VITE_API_URL=$VITE_API_URL COPY . . RUN npm run build
build-стадия — тут запекается адрес API. ARG VITE_API_URL — аргумент сборки; ENV кладёт его в окружение, и npm run build (Vite) зашивает значение прямо в бандл. Поэтому адрес API нельзя поменять в рантайме — только пересборкой. Локально берётся из .env, в проде передаётся через --build-arg (см. compose.prod.yml).
FROM nginx:alpine AS prod
Прод-стадия — на образе Nginx (веб-сервер), а не Node. Готовую статику раздаёт он. Node в проде фронту не нужен вообще.
COPY nginx.conf \ /etc/nginx/conf.d/default.conf
Кладём свой конфиг Nginx в стандартное место — переопределяем дефолтный (в нём — SPA-фолбэк на index.html).
COPY --from=build /app/dist \ /usr/share/nginx/html
Берём собранную статику из стадии build и кладём в папку, откуда Nginx раздаёт файлы по умолчанию.
CMD ["nginx", "-g", "daemon off;"]
daemon off; — запускаем Nginx на переднем плане. Контейнер живёт, пока живёт его главный процесс; уйди Nginx в фон — контейнер сразу завершится.

docker-compose.yml сервера

Compose описывает, как запускать контейнеры: какие сервисы, порты, тома, связи. Файл сервера — три сервиса: app, postgres, redis.

netroom-server/docker-compose.yml
services:
Корневой ключ: список контейнеров-сервисов стека. Всё внутри — отдельные сервисы.
app:
Сервис приложения. Имя app — оно же сетевое: другие контейнеры зовут его как app.
build: context: . target: dev
build — собрать образ из Dockerfile (в отличие от image, который тянет готовый). context: . — папка сборки (где Dockerfile и код). target: dev — собрать multi-stage только до стадии dev (с hot-reload).
command: npm run start:dev
Переопределяет CMD из Dockerfile при запуске. Здесь — тот же dev-режим, указан явно для наглядности.
ports: - "3030:3030"
Проброс порта в формате хост:контейнер. Левое число — порт на твоей машине, правое — внутри контейнера. Так localhost:3030 попадает в приложение.
env_file: .env
Загрузить переменные окружения из файла .env в контейнер (PORT, DATABASE_URL, CORS_ORIGIN и т.д.).
volumes: - .:/app
Bind-mount: папка проекта с хоста (.) монтируется в /app контейнера. Правки кода сразу видны внутри — основа hot-reload.
- /app/node_modules
Анонимный том поверх /app/node_modules: «прикрывает» node_modules из образа, чтобы хостовая папка их не затёрла. Без него — частое «module not found».
depends_on: postgres: condition: service_healthy redis: condition: service_started
depends_on — порядок и условия старта. service_healthy: app стартует, только когда postgres прошёл healthcheck (реально готов принимать соединения), а не просто «процесс запущен». Для redis достаточно service_started — что контейнер запустился.
postgres: image: postgres:18-alpine
Сервис базы. image (не build) — тянем готовый образ Postgres 18 (alpine) из Docker Hub, ничего не собираем.
environment: POSTGRES_USER: netroom POSTGRES_PASSWORD: netroom POSTGRES_DB: netroom
environment — переменные, заданные прямо здесь. Это спец-переменные образа Postgres: при первом старте он создаёт пользователя, пароль и базу с этими значениями.
ports: - "5432:5432"
Проброс порта базы на хост — чтобы подключаться к ней с машины (GUI, psql). Локально удобно; на боевом сервере этот порт наружу не открывают (урок 02).
volumes: - pgdata:/var/lib/postgresql
Именованный том pgdata монтируется туда, где Postgres 18 хранит данные. Данные переживают пересоздание контейнера. В pg18 путь — /var/lib/postgresql (без /data, как было в 17).
healthcheck: test: ["CMD-SHELL", "pg_isready -U netroom"] interval: 5s timeout: 3s retries: 5
healthcheck — как Docker проверяет, что сервис реально жив. test — команда проверки (pg_isready спрашивает Postgres, готов ли он); CMD-SHELL — выполнить через шелл. interval — как часто проверять. timeout — сколько ждать ответа. retries — сколько неудач подряд, прежде чем пометить контейнер unhealthy.
redis: image: redis:8-alpine ports: - "6379:6379" volumes: - redisdata:/data
Сервис кэша. Готовый образ Redis 8. Порт проброшен на хост, данные — в именованном томе redisdata (путь /data — стандартный для образа Redis).
volumes: pgdata: redisdata:
Объявление именованных томов (верхний уровень файла). Docker создаёт и хранит их сам, вне папки проекта. Здесь просто регистрируем имена, которые выше использовали сервисы.

docker-compose.yml клиента

Проще серверного: один сервис web, без базы и кэша. Ключи те же самые.

netroom-client/docker-compose.yml
services: web: build: context: . target: dev
Единственный сервис web. Собирается из своего Dockerfile до стадии dev (Vite dev-сервер).
command: npm run dev -- --host --port 3000
Запускает Vite так, чтобы он был виден снаружи контейнера (--host) и на порту 3000 (--port).
ports: - "3000:3000"
Фронт доступен на localhost:3000. Другой порт, чем у бэка (3030) — не конфликтуют.
env_file: .env volumes: - .:/app - /app/node_modules
Тот же приём, что у сервера: env из .env (тут VITE_API_URL), bind-mount кода для hot-reload и анонимный том под node_modules.

docker-compose.yml админки

Как клиентский, отличается двумя строками.

netroom-admin/docker-compose.yml · отличия
services: admin:
Сервис называется admin (у клиента — web). Разные имена важны, если поднимать оба в одном проекте.
ports: - "3001:3000"
хост:контейнер. Внутри контейнера Vite слушает обычный 3000 — контейнеры изолированы, конфликта нет. А на хосте отдаём 3001, потому что 3000 уже занят клиентом.

infra: compose.dev.yml — весь стек разом

Крошечный файл в репе netroom-infra. Ничего не дублирует — через include подключает compose трёх приложений.

netroom-infra/compose.dev.yml
include: - ../netroom-server/docker-compose.yml - ../netroom-client/docker-compose.yml - ../netroom-admin/docker-compose.yml
include — подключить чужие compose-файлы и слить их сервисы в один проект и общую сеть. Пути с ../ — репы лежат рядом. Запуск: docker compose -f compose.dev.yml up. Каждая репа держит свой compose, тут только ссылки.

compose.prod.yml — боевой файл

Отдельный файл для сервера (dev и prod не запускают вместе). Разбираем только то, чем он отличается от dev.

netroom-infra/compose.prod.yml · ключевые отличия
backend: build: context: ../netroom-server target: prod
Собирается из соседней репы (../netroom-server), стадия prod — собранный dist под non-root, без dev-инструментов.
environment: CORS_ORIGIN: https://app.${DOMAIN},... DATABASE_URL: postgres://netroom:${POSTGRES_PASSWORD}@postgres:5432/netroom
${DOMAIN} и ${POSTGRES_PASSWORD} Compose подставляет из .env. Так и CORS-список, и строка подключения выводятся из одного домена и пароля. Хосты БД/кэша — имена сервисов приватной сети.
restart: unless-stopped
Docker сам поднимет контейнер после падения или перезагрузки сервера. Стоит у всех сервисов прод-файла.
client: build: context: ../netroom-client target: prod args: VITE_API_URL: https://api.${DOMAIN}
args — аргументы сборки (не рантайма). VITE_API_URL уезжает в тот самый ARG Dockerfile и запекается в бандл. Admin — так же. Сменишь домен → нужен --build.
postgres: image: postgres:18-alpine # НЕТ секции ports
Главное отличие от dev. У postgres и redis нет ports — наружу не смотрят вообще. Backend достаёт их по приватной сети как postgres / redis. Открыть порт = открыть базу интернету.
nginx: image: nginx:alpine ports: - "80:80" - "443:443"
Единственный публичный сервис. Только он публикует порты. Внутри — reverse-proxy на три поддомена и TLS.
volumes: - ./nginx/templates:/etc/nginx/templates:ro - ./nginx/snippets:/etc/nginx/snippets:ro - certbot_conf:/etc/letsencrypt:ro - certbot_www:/var/www/certbot:ro
Конфиг и snippets — :ro (read-only). certbot_conf — сертификаты (их пишет certbot, nginx только читает). certbot_www — папка для ACME-проверки.
command: > envsubst '$${DOMAIN}' < ...template > ...default.conf; while :; do sleep 6h; nginx -s reload; done & exec nginx -g 'daemon off;'
envsubst '$${DOMAIN}' подставляет в шаблон только ${DOMAIN}, не трогая переменные nginx ($host, $scheme). $$ — чтобы Compose не съел $ сам. Цикл reload раз в 6ч — подхватить обновлённый сертификат.
certbot: image: certbot/certbot entrypoint: > while :; do certbot renew ...; sleep 12h; done
Отдельный сервис, каждые 12ч пробует продлить сертификат (сам certbot действует только когда срок близок). Первый выпуск — скриптом init-letsencrypt.sh.
volumes: pgdata: redisdata: certbot_conf: certbot_www:
Именованные тома. certbot_conf хранит выпущенные сертификаты — терять нельзя (у Let's Encrypt лимиты на выпуск). certbot_www — общая папка ACME между nginx и certbot.

Конфиг Nginx — ключевые строки

netroom-infra/nginx/templates/default.conf.template
map $http_upgrade $connection_upgrade { default upgrade; '' close; }
map вычисляет $connection_upgrade из заголовка Upgrade: upgrade для WebSocket, иначе close. Используется в proxy.conf.
location /.well-known/acme-challenge/ { root /var/www/certbot; }
На :80 — сюда certbot кладёт файл-проверку, Let's Encrypt его забирает. Так подтверждается, что домен реально твой.
location / { return 301 https://$host$request_uri; }
Весь остальной HTTP — редирект на HTTPS. $host/$request_uri — переменные nginx (их envsubst трогать не должен).
ssl_certificate /etc/letsencrypt/live/api.${DOMAIN}/fullchain.pem; ssl_certificate_key /etc/letsencrypt/live/api.${DOMAIN}/privkey.pem;
Один сертификат на все три поддомена (SAN), лежит в папке первого домена — api.${DOMAIN}. Поэтому все три server-блока ссылаются сюда.
location / { include /etc/nginx/snippets/proxy.conf; proxy_pass http://backend:3030; }
proxy_pass — куда переслать: backend:3030 (api), client:80 (app), admin:80 (admin) — по именам сервисов приватной сети. include proxy.conf — общие заголовки.
snippets и init-letsencrypt.sh

proxy.conf — общие заголовки прокси (X-Forwarded-*, WebSocket). ssl-params.conf — TLS по профилю Mozilla Intermediate + HSTS. А init-letsencrypt.sh запускается один раз: разрывает «курицу и яйцо» (nginx не стартует без сертификата, а сертификат не выпустить без nginx) — кладёт пустышку, поднимает nginx, выпускает настоящий, перезагружает. Подробно — в Уроке 02.

короткое правило на будущее

Dockerfile отвечает на вопрос «как собрать образ» (из чего, что скопировать, чем запускать). compose отвечает «как это запускать вместе» (порты, тома, сеть, связи между сервисами). Одно — рецепт, другое — оркестровка.
