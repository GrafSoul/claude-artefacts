Coolify — залить netroom без ручного Nginx

В уроке 02 мы поднимали прод руками: Nginx, certbot, compose.prod.yml. Coolify — панель, которая делает это за тебя: обратный прокси, авто-SSL, базы в один клик, деплой по git push. Ниже — установка и пошаговый залив всех четырёх реп netroom плюс Postgres и Redis.

Что такое Coolify и что он заменяет

Coolify — self-hosted PaaS: ставишь на свой VPS, и он превращается в подобие Vercel или Heroku, но на твоём железе и бесплатно. Под капотом — всё тот же Docker и обратный прокси Traefik, просто спрятанные за панелью. Он не отменяет то, что мы разбирали в уроках, — он это автоматизирует.

что Coolify НЕ делает за тебя

Базовую защиту сервера — всё ещё сам: непарольный SSH, ufw, fail2ban (шаги 1–2 урока 02). Coolify — про приложения, а не про хардненг ОС. И само понимание, что он делает под капотом, — тоже твоё: когда что-то отвалится, чинить будешь ты.

Установка на VPS

Нужен чистый VPS с Linux (Ubuntu/Debian и др.), 64-бит (AMD64 или ARM64). Минимум по докам — 2 CPU, 2 ГБ RAM, 30 ГБ диска (для комфорта лучше 2+ ГБ с запасом, т.к. на этом же сервере крутятся и твои контейнеры).

Зайди на сервер по SSH под root и выполни официальный скрипт:

# ставит Docker 24+, зависимости, создаёт /data/coolify и запускает панель curl -fsSL https://cdn.coollabs.io/coolify/install.sh | sudo bash

Скрипт сам поставит Docker, нужные утилиты, сгенерирует SSH-ключи и поднимет Coolify. Через пару минут открывай панель в браузере:

http://<IP-сервера>:8000
первый вход — сразу

На первом экране Coolify просит создать админ-аккаунт. Сделай это немедленно: пока аккаунта нет, регистрация открыта — любой, кто знает IP, может занять твою панель. Открыл порт 8000 — сразу заведи админа.

Модель Coolify: Project → Environment → Resource

Три уровня вложенности — важно понять сразу, дальше всё по ним раскладывается.

1Project (проект)

Логическая коробка под весь netroom. Создаёшь один проект — netroom.

2Environment (окружение)

Внутри проекта — окружения: production, при желании staging. Обычно стартуешь с production.

3Resource (ресурс)

Конкретная штука внутри окружения: приложение (из Git), база данных, сервис. Наш netroom — это 5 ресурсов в одном окружении: Postgres, Redis, server, client, admin.

Сервер, на который всё катится, подключается отдельно (раздел Servers). Сразу после установки уже есть сервер localhost — тот самый VPS, где стоит Coolify. Для старта катим прямо на него.

План: как разложить netroom

Один проект netroom, окружение production, внутри — пять ресурсов. Домены — три поддомена, как и в ручном варианте.

Пошагово: залить наше барахло

0DNS — три A-записи

У регистратора домена наведи на IP сервера: api.домен, app.домен, admin.домен → A-запись на один и тот же IP. Без этого Let's Encrypt не выпустит сертификат.

1Сервер + Coolify

VPS готов, скрипт установки выполнен (см. выше), админ создан. Сервер localhost уже подключён.

2Подключить GitHub

Раздел Sources → добавь GitHub App (проще всего): даёт доступ к приватным репам netroom-* и включает авто-деплой по push. Альтернатива для одной репы — Deploy Key.

3Project + Environment

Создай проект netroom, внутри — окружение production. Дальше все ресурсы кладём сюда.

4PostgreSQL (one-click)

+ New Resource → Databases → PostgreSQL. Версию бери 18 (как локально). Coolify сам поднимет контейнер с постоянным томом. Скопируй Postgres URL (internal) из его страницы — он понадобится серверу.

5Redis (one-click)

+ New Resource → Databases → Redis. Тоже приватный, с томом. Скопируй его internal URL.

6netroom-server (backend)

+ New Resource → Application → источник netroom-server → Build Pack Dockerfile. Порт 3030, домен https://api.домен. Переменные (Runtime): DATABASE_URL и REDIS_URL из шагов 4–5, CORS_ORIGIN=https://app.домен,https://admin.домен. Деплой.

7netroom-client

+ New Resource → Application → netroom-client → Dockerfile. Порт 80 (прод-стадия — статика в nginx), домен https://app.домен. Переменная VITE_API_URL=https://api.домен с включённым чек-боксом Build Variable (см. ниже — это критично). Деплой.

8netroom-admin

То же, что client, но репа netroom-admin и домен https://admin.домен. Порт 80, VITE_API_URL как Build Variable.

9Авто-деплой

Если ставил GitHub App — авто-деплой обычно уже включён. Проверь на каждом приложении: Advanced → Auto Deploy. Теперь git push в ветку = пересборка и выкат.

Настройки по каждому нашему ресурсу

Шпаргалка: выбери сервис — покажу точные значения полей для Coolify.

про Dockerfile и стадии

Наши Dockerfile multi-stage (base → deps → dev → build → prod). Coolify по Build Pack «Dockerfile» собирает финальную стадию — а она у нас prod. Для сервера это лёгкий Node-образ, для client/admin — nginx со статикой. Ничего дополнительно указывать не нужно. Подробнее про стадии — в Docker: ментальная модель.

Build vs Runtime — почему это важно для фронтов

В Coolify у каждой переменной два независимых флага (оба включены по умолчанию):

Build Variable

Подставляется при сборке образа — как Docker ARG. Именно сюда идёт VITE_API_URL: Vite запекает адрес API в бандл на этапе build.

Runtime Variable

Доступна внутри работающего контейнера через .env. Сюда — то, что читается при старте: DATABASE_URL, REDIS_URL, CORS_ORIGIN.

главная ловушка фронта

Если у VITE_API_URL не отмечен Build Variable — Vite не увидит его при сборке, и бандл уедет с пустым адресом API. Фронт «залился, но не видит бэкенд». Для client и admin VITE_API_URL обязан быть Build Variable; Runtime ему не нужен.

Редактировать переменные можно карточками (Normal View, там и чек-боксы Build/Multiline/ Literal) или пачкой как .env-текст (Developer View). Для секретов есть Use Docker Build Secrets (BuildKit) — значение не оседает в слоях образа.

Частые вопросы

Можно залить всё одним compose.prod.yml вместо пяти ресурсов?
Да — есть Build Pack Docker Compose: указываешь путь к compose-файлу в репе netroom-infra, и Coolify поднимает весь стек. Но домены и секреты тогда задаются через «магические» переменные (SERVICE_FQDN_*, SERVICE_PASSWORD_*), и нельзя объявлять свои сети в compose (Coolify сам делает bridge; кастомная сеть ломает роутинг Traefik). Для наглядного управления доменами и env проще пять отдельных ресурсов; compose — когда хочешь один источник правды.
Coolify заменяет то, что я учил в уроке 02? Зря учил?
Не зря. Coolify — обёртка над тем же Docker + reverse-proxy + Let's Encrypt. Пока работает — кликаешь и не думаешь. Когда сломается (сертификат не выпустился, контейнер не поднялся, база не видна) — чинить будешь ты, и тут пригодится понимание из уроков. Ручной путь — чтобы понимать, Coolify — чтобы быстро катить.
Postgres и Redis точно приватные?
Да. Базы в Coolify по умолчанию не имеют публичного домена/порта — доступны только другим ресурсам по внутреннему адресу (internal URL, который ты копируешь). Публичный доступ включается отдельно и осознанно (например, чтобы подключиться GUI-клиентом). Это ровно наш принцип «база не торчит в интернет».
Как обновлять — снова кликать?
Нет. С Auto Deploy (GitHub App или webhook) любой git push в отслеживаемую ветку запускает пересборку и выкат сам. Можно и вручную кнопкой Deploy. Откат — из истории деплоев.
Что с бэкапами базы?
У Postgres в Coolify есть Automatic DB Backups — по расписанию в S3-совместимое хранилище, восстановление в один клик. Плюс раздел Import Backups для загрузки дампов (pg_dump -Fc по умолчанию). Настраивается на странице самой базы.
