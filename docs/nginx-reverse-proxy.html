Nginx — привратник системы

На боевом сервере снаружи торчит ровно один сервис — Nginx. Он принимает все запросы, снимает с них шифрование и раздаёт по нужным адресам: app. — статика клиента, api. — на бэкенд, admin. — админка. Это и есть reverse-proxy. Ниже — что это значит, зачем так и как читается реальный конфиг netroom.

Что такое reverse-proxy

Прокси — посредник, который стоит между двумя сторонами и передаёт запросы. Разница только в том, чью сторону он представляет.

forward-proxy
прикрывает клиента

Стоит перед тобой и ходит в интернет от твоего имени (корпоративный прокси, VPN). Сервер видит прокси, а не тебя. Тут не про это.

reverse-proxy
прикрывает серверы

Стоит перед серверами. Клиент думает, что говорит с одним сайтом, а за спиной прокси — несколько сервисов. Именно так работает Nginx в netroom.

Аналогия — ресепшн в бизнес-центре. Посетитель приходит к одной стойке, называет, к кому он, и его провожают в нужный кабинет. Снаружи видно только ресепшн; сколько за ним отделов и где они — гостю знать не нужно. Nginx — эта стойка: единственный вход, за которым спрятаны client, backend и admin.

Роутинг по поддоменам

Как Nginx понимает, куда вести запрос? Смотрит на имя хоста (Host-заголовок) — то, что в адресной строке. Выбери поддомен и посмотри, куда уйдёт запрос.

TLS-терминация: где снимается шифрование

Снаружи всё идёт по HTTPS — трафик зашифрован. Расшифровывает его Nginx, на своей границе. Дальше, уже внутри приватной сети Docker, он общается с backend и статикой по обычному HTTP. Это и называется TLS-терминация — «шифрование заканчивается здесь».

Почему так удобно: сертификат и все тонкости TLS живут в одном месте, а не в каждом сервисе. Бэкенду не нужно знать про сертификаты — он просто отвечает на HTTP. Внутренняя сеть закрыта от интернета, поэтому обычного HTTP внутри достаточно.

коротко

снаружи: https://api.домен (шифр) → Nginx расшифровал → внутри: http://backend:3030 (без шифра, но в закрытой сети).

Конфиг построчно

Реальный файл nginx/templates/default.conf.template. Разберём по блокам. Три HTTPS-блока (api/app/admin) устроены одинаково — покажем один.

1 · WebSocket-мост

default.conf.template · map
map $http_upgrade $connection_upgrade { default upgrade; '' close; }
map заводит переменную $connection_upgrade на основе заголовка Upgrade. Если браузер просит поднять WebSocket (Upgrade: websocket) — значение upgrade; если заголовок пустой — close. Дальше это подставляется в proxy.conf, чтобы прокси не рвал WebSocket-соединения.

2 · HTTP на :80 — проверка сертификата и редирект

default.conf.template · server :80
server { listen 80; listen [::]:80; server_name api.${DOMAIN} app.${DOMAIN} admin.${DOMAIN};
Блок для обычного HTTP. listen 80 — слушать порт 80 (IPv4), [::]:80 — то же для IPv6. server_name — на какие домены отвечает этот блок: сразу все три поддомена.
location /.well-known/acme-challenge/ { root /var/www/certbot; }
Единственное, что отдаётся по HTTP. Сюда certbot кладёт файл-проверку, а Let's Encrypt его читает — так подтверждается, что домен твой. Папка общая с certbot через том certbot_www.
location / { return 301 https://$host$request_uri; }
Всё остальное по HTTP — 301-редирект на HTTPS. $host — запрошенный домен, $request_uri — путь. Это переменные nginx, и envsubst их не трогает (подставляет только ${DOMAIN}).

3 · HTTPS на :443 — сам прокси

default.conf.template · server :443 (api → backend)
server { listen 443 ssl; listen [::]:443 ssl; http2 on; server_name api.${DOMAIN};
HTTPS-блок. ssl — включить шифрование на этом порту. http2 on — версия протокола HTTP/2 (быстрее: мультиплексирование запросов в одном соединении). server_name — этот блок отвечает только за api.-поддомен.
ssl_certificate .../api.${DOMAIN}/fullchain.pem; ssl_certificate_key .../api.${DOMAIN}/privkey.pem;
Пути к сертификату и приватному ключу. Сертификат один на все три поддомена (SAN-сертификат), поэтому лежит в папке первого домена — api.${DOMAIN}. Блоки app и admin ссылаются сюда же.
include /etc/nginx/snippets/ssl-params.conf;
Подключить общие TLS-настройки (протоколы, шифры, security-заголовки) — чтобы не дублировать их в трёх блоках. Разберём ниже.
location / { include /etc/nginx/snippets/proxy.conf; proxy_pass http://backend:3030; } }
Сердце прокси. proxy_pass — куда переслать запрос: на сервис backend, порт 3030, по имени в приватной сети Docker. У блока app. здесь client:80, у admin. — admin:80. include proxy.conf добавляет общие заголовки.

Осторожно: тут ДВА разных Nginx

Частая путаница. Слово «nginx» в проекте встречается в двух совершенно разных ролях — не перепутай их.

уровень 1 · infra
Nginx-привратник

Отдельный сервис в compose.prod.yml. Держит TLS и роутит по поддоменам на backend / client / admin. Про него весь этот документ.

уровень 2 · внутри client / admin
Nginx-раздатчик

Мини-Nginx внутри контейнера фронта (из его Dockerfile). Просто раздаёт собранную статику и делает SPA-фолбэк на index.html. TLS и поддоменов не знает.

Поэтому proxy_pass http://client:80 — это привратник (уровень 1) стучится во внутренний Nginx контейнера client (уровень 2), который и отдаёт файлы. Два веб-сервера в цепочке, у каждого своя работа.

snippets: переиспользуемые куски

Чтобы не копировать одни и те же строки в три HTTPS-блока, повторяющееся вынесено в snippets/ и подключается через include.

proxy.conf — заголовки прокси

snippets/proxy.conf
proxy_http_version 1.1;
Общаться с сервисом по HTTP/1.1 — нужно, чтобы работали WebSocket-апгрейды (в 1.0 их нет).
proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme;
Передать бэкенду правду о клиенте. Иначе он увидит, что все запросы пришли «от nginx». Host — исходный домен; X-Real-IP / X-Forwarded-For — настоящий IP клиента; X-Forwarded-Proto — что снаружи был https (внутри-то http). Приложение читает эти заголовки, когда ему важен реальный клиент.
proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection $connection_upgrade;
Пробросить WebSocket-апгрейд. Тут и пригождается $connection_upgrade из map в начале конфига: для ws — upgrade, для обычных запросов — close.

ssl-params.conf — настройки TLS

snippets/ssl-params.conf
ssl_protocols TLSv1.2 TLSv1.3; ssl_prefer_server_ciphers off; ssl_ciphers ECDHE-...;
Профиль Mozilla «Intermediate». Разрешены только современные версии TLS (1.2 и 1.3) и безопасные наборы шифров. Старьё (TLS 1.0/1.1) отключено. Это баланс «безопасно, но совместимо с обычными браузерами».
ssl_session_cache shared:SSL:10m; ssl_session_timeout 1d; ssl_session_tickets off;
Кэш TLS-сессий: повторные подключения не проходят полное рукопожатие заново — быстрее. 10m — размер кэша, 1d — срок хранения.
add_header Strict-Transport-Security "max-age=63072000" always; add_header X-Content-Type-Options "nosniff" always; add_header X-Frame-Options "SAMEORIGIN" always; add_header Referrer-Policy "strict-origin-when-cross-origin" always;
Security-заголовки в каждый ответ. Strict-Transport-Security (HSTS) — браузер запомнит «сюда только по HTTPS» на 2 года. X-Content-Type-Options: nosniff — не угадывать тип файла. X-Frame-Options: SAMEORIGIN — не встраивать сайт в чужие iframe (защита от кликджекинга). Referrer-Policy — что слать в заголовке Referer.

Частые вопросы

Зачем Nginx-привратник, если фронт и так раздаётся Nginx внутри контейнера?
Это разные задачи. Внутренний Nginx контейнера просто отдаёт файлы одного приложения на :80 и ничего не знает про TLS, домены и другие сервисы. Привратник — единая точка входа для всего стека: он держит сертификат, расшифровывает HTTPS и решает, какому из трёх сервисов отдать запрос. Без него пришлось бы вешать TLS и роутинг на каждый сервис отдельно.
Почему один сертификат на три поддомена?
Сертификат умеет покрывать несколько доменов сразу — это поле SAN (Subject Alternative Name). certbot выпускает его на api., app. и admin. одной командой. Физически файл лежит в папке первого домена (live/api.домен/), поэтому все три server-блока указывают путь именно туда.
Почему роутинг по поддоменам, а не по путям вроде /api?
Оба варианта рабочие. Поддомены дают чистое разделение: у каждого приложения свой origin, свои куки, проще масштабировать и выносить на разные машины. Путь-роутинг (/api, /admin) экономит на DNS-записях, но смешивает всё в один origin и усложняет куки и кэш. В netroom выбрали поддомены.
Что расшифровывает HTTPS — Nginx или бэкенд?
Nginx, на входе. Дальше внутри приватной сети он идёт к бэкенду по обычному HTTP. Бэкенду не нужно возиться с сертификатами — это упрощает и его код, и обновление сертификата (меняется в одном месте). Чтобы приложение всё же знало, что снаружи был HTTPS, Nginx сообщает это заголовком X-Forwarded-Proto.
Зачем в конфиге и :80, и :443?
:443 — сам HTTPS-трафик. :80 нужен для двух вещей: принять ACME-проверку от Let's Encrypt (она идёт по HTTP) и увести на HTTPS всех, кто пришёл по голому http:// (редирект 301). Так пользователь, набравший адрес без «https», всё равно попадёт на защищённую версию.
А что если запрос придёт на неизвестный поддомен или голый IP?
Если server_name ни с чем не совпал, Nginx отдаёт запрос первому подходящему server-блоку на этом порту (в нашем файле для :443 первым идёт api.). Чтобы такие «чужие» запросы жёстко отбивать, добавляют отдельный default_server-блок, возвращающий 444/403 — в базовой сборке netroom его пока нет.
